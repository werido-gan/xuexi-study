## 1. Web开发基础（入门）

### 1.1 什么是 Web？

**通俗理解**：想象 Web 是一张无限大的蜘蛛网，每一个节点就是一个网页，连接节点的丝线就是超链接（Hyperlink）。你用浏览器点击链接，就是在这张网上"爬行"。

**技术定义**：Web（World Wide Web，万维网）是一种基于互联网运行的信息系统，通过 HTTP 协议在服务器与浏览器之间传输 HTML 文档、图片、视频等资源。

```
[Web 的核心组成]

  用户（浏览器）
       │
       │  HTTP 请求
       ▼
   互联网（Internet）
       │
       │  HTTP 响应
       ▼
   服务器（Server）
       │
   ┌───┴───┐
   │  数据  │
   │  逻辑  │
   │  存储  │
   └───────┘
```

**Web 的三大基石：**

- **HTML**：网页的骨架，描述内容结构（这是什么）
- **CSS**：网页的皮肤，描述视觉样式（长什么样）
- **JavaScript**：网页的肌肉，描述交互行为（能做什么）

**类比**：HTML 是一栋楼的钢筋混凝土结构，CSS 是装修设计，JavaScript 是电梯、自动门、空调等动态设施。

### 1.2 浏览器工作原理（通俗 + 深入）

#### 通俗版

浏览器就像一个"超级翻译官"。你输入网址，它去帮你找那个网站服务器，把服务器发来的"代码文件"（HTML、CSS、JS）翻译成你看到的漂亮网页。

#### 深入版：浏览器的五大核心模块

```
┌─────────────────────────────────────────────────┐
│                   浏览器                          │
│                                                   │
│  ┌──────────┐  ┌──────────┐  ┌──────────────┐   │
│  │  用户界面 │  │  浏览器  │  │   数据持久化  │   │
│  │  (UI层)  │  │  引擎    │  │  (Cookie/   │   │
│  │          │  │          │  │   Storage)  │   │
│  └──────────┘  └────┬─────┘  └──────────────┘   │
│                     │                             │
│               ┌─────▼──────┐                     │
│               │  渲染引擎   │ ← HTML/CSS解析      │
│               │ (Rendering │                     │
│               │  Engine)   │                     │
│               └─────┬──────┘                     │
│                     │                             │
│      ┌──────────────┼──────────────┐             │
│      │              │              │             │
│  ┌───▼────┐  ┌──────▼────┐  ┌────▼──────┐      │
│  │  JS引擎 │  │ 网络模块  │  │  UI后端   │      │
│  │(V8等)  │  │(HTTP请求) │  │ (绘制UI)  │      │
│  └────────┘  └───────────┘  └───────────┘      │
└─────────────────────────────────────────────────┘
```

**主流渲染引擎对比：**

| 浏览器 | JS引擎 | 渲染引擎 |
|-------|--------|---------|
| Chrome | V8 | Blink |
| Firefox | SpiderMonkey | Gecko |
| Safari | JavaScriptCore | WebKit |
| Edge | V8 | Blink |

> **实际开发中的坑**：同一段 CSS 代码在不同浏览器渲染引擎下可能显示不同，这就是"浏览器兼容性"问题。解决方案：使用 CSS 前缀（-webkit-, -moz-）或 Autoprefixer 工具自动处理。

### 1.3 HTTP / HTTPS（请求流程图）

**类比**：HTTP 像发明信（明信片，内容人人可见）；HTTPS 像发加密信件（只有收件人能看懂）。

#### HTTP 请求完整流程

```
浏览器                                          服务器
  │                                               │
  │  1. 输入 URL: https://www.example.com         │
  │                                               │
  │  2. DNS解析 → 获取IP: 93.184.216.34          │
  │                                               │
  │  3. TCP 三次握手建立连接                       │
  │  ──────────────SYN────────────────────────►  │
  │  ◄─────────────SYN+ACK──────────────────────  │
  │  ──────────────ACK────────────────────────►  │
  │                                               │
  │  4. TLS握手（HTTPS特有）                      │
  │  ──────── Client Hello ───────────────────►  │
  │  ◄──────── Server Hello + 证书 ──────────────  │
  │  ──────── 密钥交换 ────────────────────────►  │
  │                                               │
  │  5. HTTP请求                                  │
  │  ───── GET / HTTP/1.1 ─────────────────────► │
  │  ───── Host: www.example.com ──────────────► │
  │                                               │
  │  6. HTTP响应                                  │
  │  ◄──── HTTP/1.1 200 OK ─────────────────────  │
  │  ◄──── Content-Type: text/html ─────────────  │
  │  ◄──── <html>...</html> ────────────────────  │
  │                                               │
  │  7. 浏览器渲染页面                            │
  │                                               │
  │  8. TCP 四次挥手断开连接                      │
  ▼                                               ▼
```

#### HTTP 常用状态码

| 状态码 | 含义 | 场景 |
|-------|------|-----|
| 200 | 成功 | 正常请求 |
| 301 | 永久重定向 | 域名迁移 |
| 304 | 未修改（缓存） | 浏览器缓存命中 |
| 400 | 请求错误 | 参数格式错误 |
| 401 | 未认证 | 需要登录 |
| 403 | 禁止访问 | 权限不足 |
| 404 | 未找到 | 资源不存在 |
| 500 | 服务器内部错误 | 后端Bug |
| 502 | 网关错误 | 服务器宕机 |

#### HTTP/1.1 vs HTTP/2 vs HTTP/3

```
HTTP/1.1: 每次请求独立连接，队头阻塞
─────────────────────────────────────────
请求1 ──► 响应1
请求2 ──────────────► 响应2
请求3 ────────────────────────► 响应3

HTTP/2: 多路复用，同一连接并行传输
─────────────────────────────────────────
请求1 ──┐
请求2 ──┤─── 并行传输 ───┤──► 响应1
请求3 ──┘               └──► 响应2、3

HTTP/3: 基于QUIC（UDP），解决传输层队头阻塞
─────────────────────────────────────────
使用UDP替代TCP，连接建立更快，丢包不影响其他流
```

> **最佳实践**：现代项目应该启用 HTTP/2，它天然支持多路复用，可大幅减少请求延迟。Nginx 配置：`listen 443 ssl http2;`

### 1.4 DNS 解析过程

**类比**：DNS 就是互联网的"电话簿"。你记不住朋友的手机号（IP地址），但记得他的名字（域名），查电话簿（DNS）就能找到号码。

```
[DNS 完整解析流程]

用户输入: www.example.com

  浏览器 → 检查浏览器DNS缓存
      ↓ (未命中)
  系统 → 检查本地 hosts 文件
      ↓ (未命中)
  系统 → 查询本地DNS服务器（ISP提供）
      ↓ (未命中)
  本地DNS → 查询根域名服务器（.）
      ↓
  根服务器 → 返回 .com 顶级域名服务器地址
      ↓
  本地DNS → 查询 .com 顶级域名服务器
      ↓
  .com服务器 → 返回 example.com 权威DNS服务器地址
      ↓
  本地DNS → 查询 example.com 权威DNS服务器
      ↓
  权威DNS → 返回 www.example.com 的 IP 地址
      ↓
  本地DNS → 缓存结果 + 返回给浏览器
      ↓
  浏览器 → 使用IP地址发起HTTP连接
```

**DNS 记录类型：**

| 记录类型 | 含义 | 示例 |
|---------|------|-----|
| A | 域名 → IPv4地址 | example.com → 93.184.216.34 |
| AAAA | 域名 → IPv6地址 | example.com → 2606:2800::1 |
| CNAME | 域名 → 另一个域名 | www → example.com |
| MX | 邮件服务器 | 指向邮件服务器 |
| TXT | 文本记录 | 域名验证、SPF记录 |

> **实际开发中的坑**：DNS 解析有 TTL（生存时间），修改 DNS 后需要等 TTL 时间才能全球生效。上线新项目时要提前修改 DNS，预留生效时间。

### 1.5 前后端分离思想

#### 传统模式（服务端渲染 SSR）

```
[传统服务端渲染]

用户请求 → 服务器 → 数据库
                 ↓
            生成完整HTML
                 ↓
            返回给浏览器
            （HTML已包含数据）
```

**问题**：前端改个按钮颜色，需要后端部署；后端接口变了，前端代码也要改。**耦合太深，效率低下**。

#### 现代前后端分离模式

```
[前后端分离架构]

                        ┌─── 前端团队 ───┐
                        │   React/Vue    │
浏览器 ◄── 静态文件 ────│  构建 → CDN    │
  │                     └────────────────┘
  │
  │ API请求（JSON）
  │
  ▼
┌─── 后端团队 ────────────────────────────┐
│                                         │
│  API网关 → 业务逻辑 → 数据库            │
│            微服务A                      │
│            微服务B                      │
└─────────────────────────────────────────┘
```

**优势：**
- 前后端可以独立开发、独立部署
- 前端可以部署到 CDN，性能极佳
- 同一套 API 可以服务 Web、App、小程序
- 技术栈可以各自选择最合适的

**约定**：前后端通过 API 文档（Swagger/OpenAPI）约定接口，互不干扰。

## 2. HTML / CSS / JavaScript 核心三件套

### 2.1 HTML 结构与语义化

#### HTML 文档基本结构

```html
<!DOCTYPE html>           <!-- 告诉浏览器这是HTML5文档 -->
<html lang="zh-CN">      <!-- 根元素，lang属性指定语言 -->
  <head>
    <!-- 头部：不显示在页面上，但对浏览器和搜索引擎很重要 -->
    <meta charset="UTF-8">                    <!-- 字符编码 -->
    <meta name="viewport"                     <!-- 移动端适配关键！ -->
          content="width=device-width, initial-scale=1.0">
    <meta name="description" content="页面描述">  <!-- SEO -->
    <title>页面标题</title>                   <!-- 标签页标题 -->
    <link rel="stylesheet" href="style.css">  <!-- 引入CSS -->
  </head>
  <body>
    <!-- 页面内容 -->
    <script src="app.js"></script>  <!-- JS放在body底部，防止阻塞渲染 -->
  </body>
</html>
```

#### 语义化标签（Semantic HTML）

**为什么要语义化？**
1. 对搜索引擎友好（SEO），爬虫能理解内容结构
2. 对残障用户友好（屏幕阅读器依赖语义）
3. 代码可读性强，团队协作更容易

```
[非语义化 vs 语义化对比]

❌ 非语义化（全用div，无意义）
<div class="header">
  <div class="nav">
    <div class="nav-item">首页</div>
  </div>
</div>
<div class="main">
  <div class="article">文章内容</div>
  <div class="sidebar">侧边栏</div>
</div>
<div class="footer">版权信息</div>

✅ 语义化（标签即意义）
<header>
  <nav>
    <a href="/">首页</a>
  </nav>
</header>
<main>
  <article>文章内容</article>
  <aside>侧边栏</aside>
</main>
<footer>版权信息</footer>
```

**常用语义化标签速查：**

| 标签 | 语义 | 用途 |
|-----|------|-----|
| `<header>` | 页头 | 网站Logo、导航 |
| `<nav>` | 导航 | 菜单链接 |
| `<main>` | 主内容 | 页面核心内容（每页只用一次） |
| `<article>` | 文章 | 独立完整的内容（博客文章） |
| `<section>` | 章节 | 内容分区 |
| `<aside>` | 旁白 | 侧边栏、广告 |
| `<footer>` | 页脚 | 版权、联系方式 |
| `<figure>` | 图表 | 图片、图表及说明 |
| `<time>` | 时间 | 日期时间（有datetime属性） |
| `<address>` | 地址 | 联系信息 |

#### HTML 元素类型

```
[元素分类]

块级元素（Block）          行内元素（Inline）
独占一行，可设宽高          不独占一行，宽高由内容决定

<div>   <p>              <span>  <a>
<h1~h6> <ul/ol>          <strong> <em>
<table> <form>           <img>   <input>
<header> <section>       <button> <label>
```

> **实际开发中的坑**：`<img>` 是行内元素，底部默认有 4px 空白（基线对齐问题）。解决方法：`img { display: block; }` 或 `img { vertical-align: bottom; }`

### 2.2 CSS 布局（Flex / Grid / 响应式）

#### 盒子模型（Box Model）

**类比**：每个 HTML 元素就像一个礼品盒。内容是礼物本身，padding 是内部填充物，border 是盒子壁，margin 是盒子之间的距离。

```
[CSS 盒子模型]

┌──────────────────────────────────────┐
│              margin（外边距）         │
│  ┌────────────────────────────────┐  │
│  │          border（边框）         │  │
│  │  ┌──────────────────────────┐  │  │
│  │  │       padding（内边距）   │  │  │
│  │  │  ┌────────────────────┐  │  │  │
│  │  │  │   content（内容）   │  │  │  │
│  │  │  │   width × height   │  │  │  │
│  │  │  └────────────────────┘  │  │  │
│  │  └──────────────────────────┘  │  │
│  └────────────────────────────────┘  │
└──────────────────────────────────────┘

⚠️ 重要：box-sizing 属性
- content-box（默认）：width = 内容宽度（不含padding、border）
- border-box（推荐）：width = 内容 + padding + border
  → 所有项目建议设置：* { box-sizing: border-box; }
```

#### Flexbox 布局（一维布局）

**类比**：Flex 像一排可伸缩的弹簧座位，座位可以自动调整大小来填满空间。

```
[Flexbox 核心概念]

容器（container）设置 display: flex
  │
  ├── 主轴方向（flex-direction）
  │     row（←→ 水平，默认）
  │     column（↑↓ 垂直）
  │
  ├── 主轴对齐（justify-content）
  │     flex-start  | ● ● ● |
  │     flex-end    |       ● ● ●|
  │     center      |   ● ● ●   |
  │     space-between| ●   ●   ● |
  │     space-around| ○●○ ○●○ ○●○|
  │     space-evenly| ○ ● ○ ● ○ ● ○|
  │
  └── 交叉轴对齐（align-items）
        stretch（默认，撑满容器高度）
        center（垂直居中）
        flex-start / flex-end
```

**常用 Flex 布局示例：**

```css
/* 水平垂直居中（面试高频题） */
.container {
  display: flex;
  justify-content: center;  /* 水平居中 */
  align-items: center;      /* 垂直居中 */
}

/* 导航栏：左侧Logo，右侧菜单 */
.navbar {
  display: flex;
  justify-content: space-between;
  align-items: center;
}

/* 弹性子元素 */
.flex-item {
  flex: 1;          /* 等分剩余空间 */
  flex-grow: 1;     /* 放大比例 */
  flex-shrink: 0;   /* 不缩小 */
  flex-basis: 200px; /* 初始大小 */
}
```

#### CSS Grid 布局（二维布局）

**类比**：Grid 像一个 Excel 表格，可以精确控制行和列。

```
[Grid 布局示意]

grid-template-columns: 1fr 2fr 1fr
grid-template-rows: 100px auto 60px

┌──────────┬────────────────────┬──────────┐
│  1fr     │       2fr          │   1fr    │ ← 100px
├──────────┼────────────────────┼──────────┤
│          │                    │          │ ← auto
│  left    │      main          │  right   │
│          │                    │          │
├──────────┴────────────────────┴──────────┤
│              footer                      │ ← 60px
└──────────────────────────────────────────┘
```

```css
/* 典型页面布局 */
.layout {
  display: grid;
  grid-template-areas:
    "header header header"
    "sidebar main aside"
    "footer footer footer";
  grid-template-columns: 200px 1fr 200px;
  grid-template-rows: 60px 1fr 60px;
  min-height: 100vh;
}

.header  { grid-area: header; }
.sidebar { grid-area: sidebar; }
.main    { grid-area: main; }
.aside   { grid-area: aside; }
.footer  { grid-area: footer; }
```

#### 响应式设计

**核心思想**：同一套代码，在手机、平板、桌面上都能好看地显示。

```
[响应式断点设计]

320px ─── 480px ─── 768px ─── 1024px ─── 1440px
  │           │          │           │         │
手机竖屏    手机横屏    平板       桌面      大屏
```

```css
/* 移动优先（Mobile First）策略（推荐） */

/* 基础样式：手机 */
.container {
  width: 100%;
  padding: 0 16px;
}

/* 平板：768px以上 */
@media (min-width: 768px) {
  .container {
    max-width: 720px;
    margin: 0 auto;
  }
}

/* 桌面：1024px以上 */
@media (min-width: 1024px) {
  .container {
    max-width: 960px;
  }
}

/* 大屏：1440px以上 */
@media (min-width: 1440px) {
  .container {
    max-width: 1200px;
  }
}
```

> **最佳实践**：始终采用 Mobile First（移动端优先）策略。原因：移动端用户占全球 60%+ ；且移动端限制更多，优先考虑限制反而能做出更简洁的设计。

### 2.3 JavaScript 核心概念

#### 变量与作用域

```javascript
// 三种声明方式
var x = 1;   // 函数作用域，存在变量提升（不推荐）
let y = 2;   // 块级作用域（推荐）
const z = 3; // 块级作用域，不可重赋值（优先使用）

// 作用域链示意
function outer() {
  const outerVar = 'outer';
  
  function inner() {
    const innerVar = 'inner';
    console.log(outerVar); // ✅ 可以访问外层变量
    console.log(innerVar); // ✅ 可以访问自身变量
  }
  
  console.log(innerVar); // ❌ 不能访问内层变量
}
```

```
[作用域链示意图]

全局作用域（Global Scope）
  └── window.x = 1
  
  函数作用域（outer）
    └── outerVar = 'outer'
    
    函数作用域（inner）
      └── innerVar = 'inner'
      
查找变量时：inner → outer → global（向上查找，不能向下）
```

#### 闭包（Closure）

**类比**：闭包就像一个"带记忆的函数"。函数执行完了，但它记住了自己诞生时的环境。

```javascript
// 经典闭包示例：计数器
function createCounter() {
  let count = 0;  // 这个变量被"封闭"在闭包中
  
  return {
    increment: () => ++count,
    decrement: () => --count,
    getCount: () => count
  };
}

const counter = createCounter();
counter.increment(); // 1
counter.increment(); // 2
counter.decrement(); // 1
counter.getCount();  // 1

// count 变量外部无法直接访问，实现了"私有变量"
```

**闭包的实际应用场景：**

```javascript
// 1. 函数工厂（生产不同配置的函数）
function multiplyBy(n) {
  return (x) => x * n;  // n 被闭包捕获
}
const double = multiplyBy(2);
const triple = multiplyBy(3);
double(5); // 10
triple(5); // 15

// 2. 防抖（Debounce）- 防止频繁触发
function debounce(fn, delay) {
  let timer;
  return function(...args) {
    clearTimeout(timer);
    timer = setTimeout(() => fn.apply(this, args), delay);
  };
}

// 3. 模块化（在ES6之前）
const myModule = (function() {
  const private = 'private data';
  
  return {
    getPrivate: () => private  // 外部只能通过这个方法访问
  };
})();
```

> **实际开发中的坑**：闭包会导致内存泄漏！如果闭包引用了大对象且长期不释放，内存会一直被占用。解决方案：不用时将引用变量设为 null。

#### 原型与原型链

**类比**：原型就像"遗传基因"。孩子（对象）会继承父母（原型）的特征（方法）。

```
[原型链示意图]

let arr = [1, 2, 3]

arr 对象
  │
  │ __proto__（原型指针）
  ▼
Array.prototype
  ├── push()
  ├── pop()
  ├── map()
  └── filter()
  │
  │ __proto__
  ▼
Object.prototype
  ├── toString()
  ├── hasOwnProperty()
  └── valueOf()
  │
  │ __proto__
  ▼
  null（原型链终点）
```

```javascript
// 原型继承实现
function Animal(name) {
  this.name = name;
}
Animal.prototype.speak = function() {
  return `${this.name} makes a sound`;
};

function Dog(name, breed) {
  Animal.call(this, name);  // 调用父类构造函数
  this.breed = breed;
}
Dog.prototype = Object.create(Animal.prototype);
Dog.prototype.constructor = Dog;
Dog.prototype.bark = function() {
  return `${this.name} barks!`;
};

// 现代写法：ES6 Class（本质还是原型）
class Animal {
  constructor(name) {
    this.name = name;
  }
  speak() {
    return `${this.name} makes a sound`;
  }
}

class Dog extends Animal {
  constructor(name, breed) {
    super(name);           // 调用父类构造函数
    this.breed = breed;
  }
  bark() {
    return `${this.name} barks!`;
  }
}
```

#### 异步编程（非常重要！）

**类比**：同步就像去餐厅点餐，站在柜台等厨师做好；异步就像下单后拿号，去找位置坐，等叫号再去取餐。

```
[同步 vs 异步]

同步（阻塞）:
任务1 ── 任务2 ── 任务3 ──► 时间
(任务1完成才能开始任务2)

异步（非阻塞）:
任务1 ──────────────────► 完成回调
任务2 ─────────► 完成回调
任务3 ──► 完成回调
         ↑
       不等待，并行执行
```

**异步的三个时代：**

```javascript
// === 时代1: 回调函数（Callback）===
// 问题：回调地狱，代码难以维护
fs.readFile('a.txt', (err, dataA) => {
  fs.readFile('b.txt', (err, dataB) => {
    fs.readFile('c.txt', (err, dataC) => {
      // 横向增长，难以阅读
    });
  });
});

// === 时代2: Promise ===
// 优点：链式调用，避免嵌套
fetch('/api/user')
  .then(res => res.json())
  .then(user => fetch(`/api/posts/${user.id}`))
  .then(res => res.json())
  .then(posts => console.log(posts))
  .catch(err => console.error(err));  // 统一错误处理

// === 时代3: async/await（推荐）===
// 优点：像写同步代码一样写异步，可读性极强
async function getUserPosts(userId) {
  try {
    const userRes = await fetch(`/api/user/${userId}`);
    const user = await userRes.json();
    
    const postsRes = await fetch(`/api/posts/${user.id}`);
    const posts = await postsRes.json();
    
    return posts;
  } catch (error) {
    console.error('获取数据失败:', error);
    throw error;
  }
}

// 并行请求（不要串行等待可以并行的请求）
async function getMultipleData() {
  // ❌ 错误写法：串行，浪费时间
  const user = await fetchUser();
  const config = await fetchConfig();  // 等user完成才开始
  
  // ✅ 正确写法：并行，节省时间
  const [user, config] = await Promise.all([
    fetchUser(),
    fetchConfig()
  ]);
}
```

> **实际开发中的坑**：`await` 只能在 `async` 函数内使用。忘记 `await` 会拿到 Promise 对象而不是值，这是非常常见的Bug！

## 3. 浏览器与网络底层原理（进阶）

### 3.1 浏览器渲染流程（Critical Rendering Path）

**类比**：浏览器渲染就像建房子。先搭骨架（DOM），再制定设计图（CSSOM），再合并成完整方案（Render Tree），再测量精确位置（Layout），最后刷漆完工（Paint）。

```
[关键渲染路径（Critical Rendering Path）]

HTML字节流
    │
    ▼
词法分析（Tokenization）→ Token流
    │
    ▼
构建 DOM 树（Document Object Model）
    │
    │                 CSS字节流
    │                     │
    │                     ▼
    │              构建 CSSOM 树
    │                     │
    └──────────┬──────────┘
               │
               ▼
         合并 Render Tree
        （只包含可见节点，
          display:none 的不包含）
               │
               ▼
           Layout（布局/回流）
         计算每个元素的精确位置和大小
               │
               ▼
         Paint（绘制/重绘）
         将像素绘制到屏幕上
               │
               ▼
         Composite（合成）
         GPU将各层合并输出到屏幕
```

**重要概念：渲染阻塞**

```
HTML 解析遇到 <script>（无defer/async）→ 暂停解析 → 下载并执行JS → 继续解析
CSS 文件 → 阻塞渲染（CSSOM构建完才能构建Render Tree）

[script 加载策略]

<script src="a.js"></script>        普通：阻塞HTML解析
                                     ─解析─|下载+执行|─继续解析─

<script defer src="a.js"></script>  defer：不阻塞解析，HTML解析完再执行
                                     ─解析─────────────────|执行|─DOMContentLoaded

<script async src="a.js"></script>  async：下载不阻塞，下载完立即执行
                                     ─解析─|执行|─────────── (执行时机不确定)
```

> **最佳实践**：
> - `<script>` 放在 `</body>` 前（防止阻塞）
> - 或使用 `defer` 属性
> - CSS `<link>` 放在 `<head>` 中（尽早加载）

### 3.2 回流与重绘

**类比**：
- **回流（Reflow）** = 重新装修，要重新测量、移家具。代价极高。
- **重绘（Repaint）** = 重新刷漆，位置没变，只是颜色变了。代价较低。

```
[触发回流的操作（代价大）]
- 改变元素尺寸（width/height/padding/margin）
- 改变元素位置（top/left）
- 添加/删除DOM元素
- 改变字体大小
- 窗口resize
- 读取布局属性（offsetTop/scrollTop/clientWidth...）
  → 浏览器为了返回准确值，会强制触发回流！

[触发重绘的操作（代价小）]
- 改变颜色（color/background-color）
- 改变visibility
- 改变outline
```

**性能优化技巧：**

```javascript
// ❌ 错误：多次读写布局属性，每次都触发回流
for (let i = 0; i < 1000; i++) {
  element.style.left = element.offsetLeft + 1 + 'px';
  // offsetLeft 强制回流 → style.left 触发回流 → 循环1000次 = 2000次回流
}

// ✅ 正确：批量读取，批量写入
const startLeft = element.offsetLeft;  // 读取一次
for (let i = 0; i < 1000; i++) {
  element.style.left = startLeft + i + 1 + 'px';
}

// ✅ 更好：使用 CSS transform，不触发回流
element.style.transform = `translateX(${startLeft + 1000}px)`;

// ✅ 使用 DocumentFragment 批量操作DOM
const fragment = document.createDocumentFragment();
for (let i = 0; i < 100; i++) {
  const li = document.createElement('li');
  li.textContent = `Item ${i}`;
  fragment.appendChild(li);  // 操作内存中的fragment，不触发回流
}
list.appendChild(fragment);  // 只触发一次回流
```

### 3.3 事件循环（Event Loop）⭐ 面试必考

**为什么 JavaScript 是单线程的？**

JS 最初设计用于浏览器，如果多线程同时操作 DOM，会产生竞争条件（Race Condition）。所以 Brendan Eich 设计时选择了单线程。但单线程如何处理异步？这就是事件循环（Event Loop）。

```
[事件循环完整图解]

    ┌─────────────────────────────────────┐
    │           调用栈（Call Stack）        │
    │   同步代码在这里执行                  │
    │                                     │
    │  ┌───────┐                          │
    │  │console│                          │
    │  ├───────┤                          │
    │  │ fn2() │                          │
    │  ├───────┤                          │
    │  │ fn1() │                          │
    │  ├───────┤                          │
    │  │main() │                          │
    │  └───────┘                          │
    └────────────────┬────────────────────┘
                     │ 调用栈为空时
                     │ Event Loop检查
                     ▼
    ┌────────────────────────────────────┐
    │          消息队列（Task Queue）      │
    │                                    │
    │  微任务队列（Microtask Queue）       │
    │  优先级更高！每次宏任务后先清空      │
    │  ┌─────────────────────────────┐   │
    │  │ Promise.then / queueMicrotask│   │
    │  │ MutationObserver             │   │
    │  └─────────────────────────────┘   │
    │                                    │
    │  宏任务队列（Macrotask Queue）       │
    │  ┌─────────────────────────────┐   │
    │  │ setTimeout / setInterval    │   │
    │  │ I/O 事件                    │   │
    │  │ requestAnimationFrame       │   │
    │  └─────────────────────────────┘   │
    └────────────────────────────────────┘
                     │
    ┌────────────────────────────────────┐
    │           Web APIs                 │
    │  浏览器提供的异步能力               │
    │  (setTimeout/fetch/DOM事件...)     │
    └────────────────────────────────────┘
```

**事件循环执行顺序：**

```javascript
console.log('1');  // 同步

setTimeout(() => {
  console.log('2');  // 宏任务
}, 0);

Promise.resolve().then(() => {
  console.log('3');  // 微任务
});

console.log('4');  // 同步

// 输出顺序: 1 → 4 → 3 → 2
// 原因: 
// 1. 同步: 1, 4
// 2. 微任务（清空）: 3
// 3. 宏任务: 2
```

**经典面试题解析：**

```javascript
async function async1() {
  console.log('async1 start');    // 2
  await async2();
  console.log('async1 end');      // 6（微任务）
}

async function async2() {
  console.log('async2');          // 3
}

console.log('script start');     // 1

setTimeout(() => {
  console.log('setTimeout');     // 8（宏任务）
}, 0);

async1();

new Promise((resolve) => {
  console.log('promise1');        // 4（Promise构造函数同步执行）
  resolve();
}).then(() => {
  console.log('promise2');        // 7（微任务）
});

console.log('script end');       // 5

// 输出: script start → async1 start → async2 → promise1 
//       → script end → async1 end → promise2 → setTimeout
```

> **理解关键**：`await` 后面的代码相当于放入微任务队列。执行到 `await async2()` 时，先执行 `async2()`，然后 `async1 end` 被放入微任务队列，继续执行后续同步代码。

### 3.4 TCP/IP 原理简述

```
[网络分层模型]

应用层   │ HTTP / HTTPS / FTP / DNS
         │ 负责：应用间通信协议
─────────┼────────────────────────
传输层   │ TCP / UDP
         │ 负责：端到端可靠传输
         │ TCP：三次握手，保证可靠
         │ UDP：不保证可靠，但速度快
─────────┼────────────────────────
网络层   │ IP / ICMP
         │ 负责：路由寻址（IP地址）
─────────┼────────────────────────
数据链路 │ Ethernet / WiFi
层       │ 负责：相邻节点间传输
─────────┼────────────────────────
物理层   │ 电缆 / 光纤 / 无线电波
         │ 负责：0和1的物理传输
```

**TCP 三次握手：**

```
客户端                    服务器
  │── SYN（我想连接你）──►  │
  │◄── SYN+ACK（好的，我也想连接你）──  │
  │── ACK（好的，连接建立）──► │
  │                         │
  │  开始传输数据              │
```

#### Vite（现代构建工具，推荐新项目使用）

**为什么 Vite 比 Webpack 快？**

```
[Webpack 开发模式]

启动时：分析所有文件 → 打包成bundle → 启动服务器
修改代码：重新打包整个bundle（文件多时非常慢）

[Vite 开发模式]

启动时：直接启动服务器（极速！）
修改代码：只更新修改的模块（HMR）

原理：浏览器原生支持ES Module，Vite直接让浏览器加载源文件
不需要打包！开发时实现"按需编译"

[Vite 为什么快？]

传统打包：
源码 → Webpack打包 → bundle.js → 浏览器

Vite方式：
源码 → Vite服务器 → 浏览器直接请求各个模块 → 按需编译
```

#### React Hooks 全解

**为什么 Hooks 出现？**

Class 组件的问题：
- 生命周期分散逻辑（componentDidMount/componentDidUpdate 里可能有相关逻辑）
- `this` 让 JavaScript 初学者困惑
- 高阶组件（HOC）和 Render Props 导致组件嵌套地狱

**核心 Hooks：**

```jsx
import { useState, useEffect, useCallback, useMemo, useRef, 
         useContext, useReducer } from 'react';

// === useState: 状态管理 ===
function Counter() {
  const [count, setCount] = useState(0);
  
  return (
    <button onClick={() => setCount(prev => prev + 1)}>
      Count: {count}
    </button>
  );
}

// === useEffect: 副作用（数据获取、订阅、DOM操作）===
function UserProfile({ userId }) {
  const [user, setUser] = useState(null);
  
  useEffect(() => {
    // 组件挂载 或 userId变化时执行
    let cancelled = false;
    
    fetchUser(userId).then(data => {
      if (!cancelled) setUser(data);  // 防止竞态条件
    });
    
    return () => {                    // 清理函数（组件卸载时执行）
      cancelled = true;
    };
  }, [userId]);                       // 依赖数组：userId变化才重新执行
  
  if (!user) return <div>Loading...</div>;
  return <div>{user.name}</div>;
}

// === useCallback: 缓存函数引用，防止子组件不必要重渲染 ===
function Parent() {
  const [count, setCount] = useState(0);
  
  // 没有useCallback：每次Parent渲染，handleClick都是新函数
  // 导致依赖它的子组件重渲染
  const handleClick = useCallback(() => {
    console.log('clicked');
  }, []);  // 依赖为空，函数永远是同一个引用
  
  return <Child onClick={handleClick} />;
}

// === useMemo: 缓存计算结果 ===
function ExpensiveComponent({ data }) {
  // 每次渲染都重新计算 → 浪费
  // const result = expensiveCalculation(data);
  
  // 只有 data 变化时才重新计算
  const result = useMemo(() => expensiveCalculation(data), [data]);
  
  return <div>{result}</div>;
}

// === useRef: 获取DOM引用 或 存储不触发重渲染的值 ===
function TextInput() {
  const inputRef = useRef(null);
  
  const focusInput = () => {
    inputRef.current.focus();  // 直接操作DOM
  };
  
  // 存储上一次的值（不触发重渲染）
  const prevCountRef = useRef(0);
  useEffect(() => {
    prevCountRef.current = count;
  });
  
  return <input ref={inputRef} />;
}

// === useReducer: 复杂状态管理 ===
const initialState = { count: 0, error: null };

function reducer(state, action) {
  switch (action.type) {
    case 'INCREMENT': return { ...state, count: state.count + 1 };
    case 'DECREMENT': return { ...state, count: state.count - 1 };
    case 'RESET':     return initialState;
    default: throw new Error(`Unknown action: ${action.type}`);
  }
}

function Counter() {
  const [state, dispatch] = useReducer(reducer, initialState);
  
  return (
    <>
      <p>Count: {state.count}</p>
      <button onClick={() => dispatch({ type: 'INCREMENT' })}>+</button>
      <button onClick={() => dispatch({ type: 'DECREMENT' })}>-</button>
    </>
  );
}
```

```jsx
// 自定义 Hook：数据获取
function useFetch(url) {
  const [data, setData] = useState(null);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);
  
  useEffect(() => {
    setLoading(true);

      .then(res => res.json())
      .then(data => {
        setData(data);
        setLoading(false);
      })
      .catch(err => {
        setError(err);
        setLoading(false);
      });
  }, [url]);
  
  return { data, loading, error };
}

// 使用
function UserList() {

  if (loading) return <Spinner />;

}
```

#### React Fiber 架构

**为什么需要 Fiber？**

React 15 的问题：虚拟 DOM 的 Diff 和更新是**同步递归**的，一旦开始就无法中断。如果组件树很大，会阻塞主线程，导致页面卡顿（掉帧）。

```
[React 15 Stack Reconciler 问题]

Diff 开始
  └── 递归整棵树（不可中断）
       └── 如果树很大，需要100ms
           └── 用户点击、动画 → 全部卡住

[React 16+ Fiber Reconciler 解决方案]

Fiber = 将工作分解成小单元（每个组件是一个Fiber节点）
     = 支持暂停、恢复、中断

渲染分两阶段：
1. Reconciliation（协调，可中断）
   └── 构建 Fiber 树
   └── 计算哪些需要更新
   └── 利用浏览器空闲时间（requestIdleCallback思想）

2. Commit（提交，不可中断）
   └── 将变更应用到真实DOM
   └── 执行生命周期/副作用
```

### 5.2 Vue 响应式原理

#### Vue 2 vs Vue 3 响应式对比

```
[Vue 2 响应式原理（Object.defineProperty）]

data: { count: 0 }
    │
    ▼  Vue2 初始化时遍历所有属性
Object.defineProperty(obj, 'count', {
  get() { 收集依赖（Watcher） },
  set(newVal) { 通知更新（Dep.notify） }
})

问题：
1. 新增属性无法检测（需要 Vue.set）
2. 数组下标修改无法检测（需要 splice 等）
3. 初始化需要递归遍历所有属性，性能有损耗

[Vue 3 响应式原理（Proxy）]

const state = reactive({ count: 0 })
    │
    ▼  使用 ES6 Proxy 代理整个对象
new Proxy(target, {
  get(target, key) { 
    track(target, key)  // 收集依赖
    return target[key] 
  },
  set(target, key, value) {
    target[key] = value
    trigger(target, key) // 触发更新
    return true
  }
})

优势：
1. 可以检测新增属性
2. 可以检测数组下标变化
3. 懒代理：访问时才代理，性能更好
```

#### Vue 3 Composition API

```javascript
// Vue 3 组合式 API
import { ref, reactive, computed, watch, onMounted } from 'vue';

export default {
  setup() {
    // ref: 基本类型响应式
    const count = ref(0);
    
    // reactive: 对象响应式
    const user = reactive({
      name: '张三',
      age: 25
    });
    
    // computed: 计算属性
    const doubleCount = computed(() => count.value * 2);
    
    // watch: 监听变化
    watch(count, (newVal, oldVal) => {
      console.log(`count: ${oldVal} → ${newVal}`);
    });
    
    // 生命周期
    onMounted(() => {
      console.log('组件已挂载');
    });
    
    const increment = () => count.value++;
    
    return { count, user, doubleCount, increment };
  }
};

// 使用 <script setup>（更简洁，推荐）
<script setup>
import { ref, computed } from 'vue';

const count = ref(0);
const doubleCount = computed(() => count.value * 2);
const increment = () => count.value++;
</script>
```

### 6.1 组件化设计

```
        └── Table
```

**组件设计原则：**

```
[好的组件设计原则]

1. 单一职责（SRP）
   每个组件只做一件事
   ❌ UserCard（既显示用户信息，又处理购物车，又发请求）
   ✅ UserCard（只负责显示用户信息）

2. 开放封闭（OCP）
   对扩展开放，对修改封闭
   通过 Props/Slots 让组件可配置，而不是修改源码

3. 受控 vs 非受控
   受控组件：数据由父组件控制（通过props传入，onChange回调）
   非受控组件：数据由自身管理（useRef读取值）

4. 容器组件 vs 展示组件（智能/傻瓜）
   容器组件：处理数据、状态、API请求
   展示组件：只管UI渲染，数据通过props传入
```

**组件通信模式：**

```
[组件通信方式]

父 → 子: Props（向下传递数据）
子 → 父: 回调函数/事件（向上传递事件）
兄弟间: 状态提升（lift state up）到共同父组件
跨层级: Context / 状态管理库（Redux/Pinia）
```

```javascript
class EventBus {
  constructor() {
    this.listeners = new Map();
  }
  
  on(event, callback) {
    if (!this.listeners.has(event)) {
      this.listeners.set(event, []);
    }
    this.listeners.get(event).push(callback);
  }
  
  emit(event, data) {
    this.listeners.get(event)?.forEach(cb => cb(data));
  }
  
  off(event, callback) {
    const cbs = this.listeners.get(event);
    if (cbs) {
      this.listeners.set(event, cbs.filter(cb => cb !== callback));
    }
  }
}
```

## 7. 性能优化（详细）

### 7.1 Web Vitals 核心指标

Google 定义的用户体验关键指标：

```
[Core Web Vitals]

LCP（Largest Contentful Paint，最大内容绘制）
  衡量：页面主要内容加载速度
  目标：≤ 2.5秒（良好）
  优化：优先加载关键资源，图片懒加载

FID（First Input Delay，首次输入延迟）
  衡量：用户第一次交互到浏览器响应的时间
  目标：≤ 100ms
  优化：减少主线程阻塞，代码分割

CLS（Cumulative Layout Shift，累计布局偏移）
  衡量：页面视觉稳定性（元素跳动）
  目标：≤ 0.1
  优化：图片/广告预留空间，避免动态内容插入

INP（Interaction to Next Paint，新增）
  衡量：整体响应性（替代FID）
  目标：≤ 200ms
```

### 7.2 首屏优化

```
[首屏时间线]

导航开始 → DNS → TCP → TLS → 请求 → 响应 → HTML解析 → CSS/JS → 渲染 → 可交互

优化每个环节：

1. 减少 DNS 查询: dns-prefetch
   <link rel="dns-prefetch" href="//cdn.example.com">

2. 提前建立连接: preconnect
   <link rel="preconnect" href="//api.example.com">

3. HTTP/2 + CDN: 并行请求 + 就近分发

4. 关键 CSS 内联: 避免首屏渲染等待CSS文件加载
   <style>/* 首屏关键CSS直接内联 */</style>

5. SSR / SSG: 服务端渲染，直接返回完整HTML
```

### 7.5 缓存策略

```
[浏览器缓存层次]

Service Worker缓存（最高优先级，可编程控制）
    │
    ▼
内存缓存（Memory Cache，关闭标签页失效）
    │
    ▼
磁盘缓存（Disk Cache，根据HTTP头决定）
    │
    ▼
网络请求

[HTTP 缓存策略]

强缓存（不发请求直接用缓存）:
Cache-Control: max-age=31536000  → 缓存1年（适合带hash的静态资源）
Cache-Control: no-cache          → 每次都要验证（不是不缓存！）
Cache-Control: no-store          → 不缓存（敏感数据）

协商缓存（发请求验证是否可用缓存）:
ETag: "abc123"                   → 内容hash
Last-Modified: Mon, 1 Jan 2024   → 最后修改时间
→ 服务器返回304 Not Modified，浏览器使用本地缓存
```

```
[缓存策略最佳实践]

静态资源（JS/CSS/图片）:
  文件名包含contenthash（如 main.abc123.js）
  Cache-Control: max-age=31536000 (1年)
  内容改变 → 文件名变 → 缓存失效

HTML文件:
  Cache-Control: no-cache（每次验证，确保拿到最新入口）

API响应:
  敏感数据: Cache-Control: no-store
  公共数据: Cache-Control: max-age=300 (5分钟)
```

## 8. 前端安全

### 8.1 XSS（跨站脚本攻击）

**类比**：XSS 就像有人在你的留言本上偷偷夹了一张会"自动执行任务"的纸条，任何看到这张纸条的人都会被影响。

```
[XSS 攻击类型]

存储型XSS（最危险）:
  攻击者 → 提交含恶意脚本的评论
  → 存入数据库
  → 其他用户访问页面
  → 浏览器执行脚本（窃取Cookie、冒充用户操作）

反射型XSS:
  攻击者 → 构造含脚本的URL
  → 诱导受害者点击
  → 服务器将URL参数反射到页面
  → 浏览器执行脚本

DOM型XSS:
  前端代码: document.innerHTML = location.hash
  → 攻击者构造 /#<img src=x onerror=alert(document.cookie)>
  → 浏览器执行脚本
```

**防御措施：**

```javascript
// 1. 对输出进行HTML转义（最关键！）
function escapeHtml(str) {
  return str
    .replace(/&/g, '&amp;')
    .replace(/</g, '&lt;')
    .replace(/>/g, '&gt;')
    .replace(/"/g, '&quot;')
    .replace(/'/g, '&#039;');
}

// 2. 使用安全的API（不用innerHTML）
// ❌ 危险
element.innerHTML = userInput;

// ✅ 安全
element.textContent = userInput;  // 自动转义

// 3. CSP（Content Security Policy）内容安全策略
// 在HTTP响应头设置
Content-Security-Policy: 
  default-src 'self';
  script-src 'self' 'nonce-xxx';  // 只允许有nonce的脚本
  style-src 'self' 'unsafe-inline';

// 4. Cookie 设置 HttpOnly（JS无法读取Cookie）
Set-Cookie: session=xxx; HttpOnly; Secure; SameSite=Strict
```

### 8.2 CSRF（跨站请求伪造）

**类比**：你已经登录了银行网站，攻击者诱导你访问他的恶意网站，该网站悄悄向银行发送了转账请求（利用你的登录状态）。

```
[CSRF 攻击流程]

1. 用户登录 bank.com（获得Cookie）
2. 用户访问 evil.com
3. evil.com 页面有: <img src="bank.com/transfer?to=hacker&amount=10000">
4. 浏览器自动带上 bank.com 的 Cookie 发送请求
5. 银行服务器认为是合法请求，转账成功！
```

**防御措施：**

```javascript
// 1. CSRF Token（主流方案）
// 服务器生成随机token，存在Session中，返回给前端
// 前端每次请求携带这个token
// 服务器验证token是否匹配

// Axios 拦截器自动加CSRF Token
axios.interceptors.request.use(config => {
  const token = getCsrfToken();  // 从Cookie或meta标签读取
  config.headers['X-CSRF-Token'] = token;
  return config;
});

// 2. SameSite Cookie（现代浏览器首选）
Set-Cookie: session=xxx; SameSite=Strict
// Strict: 第三方网站请求完全不带Cookie（最安全，但可能影响体验）
// Lax: 大多数情况不带，GET导航除外
// None: 都带（需同时设置Secure，即HTTPS）

// 3. 验证 Referer / Origin
// 服务器检查请求来源是否是合法域名
if (request.headers.origin !== 'https://bank.com') {
  return 403;
}

// 4. 敏感操作要求二次验证（密码/短信）
```

#### 商品列表页（含虚拟滚动）

```typescript
// src/pages/ProductList/index.tsx
import { useState, useCallback } from 'react';
import { useInfiniteQuery } from 'react-query';
import { useVirtualizer } from '@tanstack/react-virtual';
import { ProductCard } from '@/components/business/ProductCard';
import { useDebounce } from '@/hooks/useDebounce';
import { fetchProducts } from '@/api/product';

interface FilterState {
  keyword: string;
  categoryId: string;
  sortBy: 'price_asc' | 'price_desc' | 'sales' | 'newest';
  priceRange: [number, number];
}

export default function ProductListPage() {
  const [filter, setFilter] = useState<FilterState>({
    keyword: '',
    categoryId: '',
    sortBy: 'newest',
    priceRange: [0, 99999]
  });
  
  // 防抖处理搜索输入
  const debouncedKeyword = useDebounce(filter.keyword, 500);
  
  // 无限滚动 + 请求缓存
  const {
    data,
    fetchNextPage,
    hasNextPage,
    isFetchingNextPage,
    isLoading
  } = useInfiniteQuery({
    queryKey: ['products', { ...filter, keyword: debouncedKeyword }],
    queryFn: ({ pageParam = 1 }) => fetchProducts({
      page: pageParam,
      ...filter,
      keyword: debouncedKeyword
    }),
    getNextPageParam: (lastPage, pages) =>
      lastPage.hasMore ? pages.length + 1 : undefined,
    staleTime: 5 * 60 * 1000  // 5分钟内不重新请求
  });
  
  const products = data?.pages.flatMap(p => p.items) ?? [];
  
  // 虚拟滚动（只渲染可视区域内的DOM）
  const parentRef = useRef<HTMLDivElement>(null);
  const virtualizer = useVirtualizer({
    count: products.length,
    getScrollElement: () => parentRef.current,
    estimateSize: () => 300,  // 每项预估高度
  });
  
  return (
    <div className="product-list-page">
      <SearchBar 
        value={filter.keyword}
        onChange={keyword => setFilter(prev => ({ ...prev, keyword }))}
      />
      
      <div ref={parentRef} style={{ height: '800px', overflow: 'auto' }}>
        <div style={{ height: virtualizer.getTotalSize() }}>
          {virtualizer.getVirtualItems().map(virtualItem => (
            <div
              key={virtualItem.key}
              style={{
                position: 'absolute',
                top: virtualItem.start,
                width: '100%'
              }}
            >
              <ProductCard product={products[virtualItem.index]} />
            </div>
          ))}
        </div>
      </div>
      
      {hasNextPage && (
        <button onClick={() => fetchNextPage()} disabled={isFetchingNextPage}>
          {isFetchingNextPage ? '加载中...' : '加载更多'}
        </button>
      )}
    </div>
  );
}
```

#### 购物车状态管理

```typescript
// src/store/useCartStore.ts
import { create } from 'zustand';
import { persist } from 'zustand/middleware';  // 持久化到localStorage

interface CartItem {
  id: string;
  name: string;
  price: number;
  quantity: number;
  imageUrl: string;
  skuId: string;
}

interface CartStore {
  items: CartItem[];
  
  // 派生数据（computed）
  totalCount: () => number;
  totalPrice: () => number;
  
  // 操作方法
  addItem: (item: Omit<CartItem, 'quantity'>) => void;
  removeItem: (skuId: string) => void;
  updateQuantity: (skuId: string, quantity: number) => void;
  clearCart: () => void;
}

const useCartStore = create<CartStore>()(
  persist(
    (set, get) => ({
      items: [],
      
      totalCount: () => get().items.reduce((sum, item) => sum + item.quantity, 0),
      totalPrice: () => get().items.reduce((sum, item) => sum + item.price * item.quantity, 0),
      
      addItem: (newItem) => set(state => {
        const existing = state.items.find(item => item.skuId === newItem.skuId);
        if (existing) {
          return {
            items: state.items.map(item =>
              item.skuId === newItem.skuId
                ? { ...item, quantity: item.quantity + 1 }
                : item
            )
          };
        }
        return { items: [...state.items, { ...newItem, quantity: 1 }] };
      }),
      
      removeItem: (skuId) => set(state => ({
        items: state.items.filter(item => item.skuId !== skuId)
      })),
      
      updateQuantity: (skuId, quantity) => set(state => ({
        items: quantity <= 0
          ? state.items.filter(item => item.skuId !== skuId)
          : state.items.map(item =>
              item.skuId === skuId ? { ...item, quantity } : item
            )
      })),
      
      clearCart: () => set({ items: [] })
    }),
    {
      name: 'cart-storage',              // localStorage key
      partialize: state => ({ items: state.items })  // 只持久化items
    }
  )
);

export default useCartStore;
```

```yaml
      - uses: actions/checkout@v3
      - run: pnpm install --frozen-lockfile
      - run: pnpm build
      - uses: actions/upload-artifact@v3

          name: dist

          name: dist
```

- **初级工程师**：能实现需求，代码能跑
