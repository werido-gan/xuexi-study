## 第1章：数据结构是什么？——绪论

### 🤔 为什么要学数据结构？

想象一下，你要存储全校10000名同学的信息：

- 方案A：随机存放，查找时一个一个翻，找到要找的人平均要翻5000次
- 方案B：按学号排好序存放，用二分查找，最多翻13次就找到了！

**数据结构**就是研究"如何组织和存储数据，让我们的操作更高效"的学问。

---

### 🔑 三个最基础的概念

**概念1：数据元素**
> 数据的基本单位，就像一条完整的"记录"

例如：一名同学的完整信息（学号+姓名+成绩）就是一个**数据元素**

**概念2：数据项**
> 数据元素的组成部分，是最小的、有意义的数据单位

例如：上面那名同学的"学号"就是一个**数据项**，"姓名"也是一个**数据项**

**概念3：数据对象**
> 具有相同类型的数据元素的集合

例如：全校所有同学的信息构成一个**数据对象**

```
数据对象（全校学生信息）
    │
    ├── 数据元素（张三的全部信息：{2024001, 张三, 95分}）
    │       ├── 数据项：学号 = 2024001
    │       ├── 数据项：姓名 = 张三
    │       └── 数据项：成绩 = 95
    │
    └── 数据元素（李四的全部信息：{2024002, 李四, 88分}）
            ├── 数据项：学号 = 2024002
            └── ...
```

**数据结构 = 逻辑结构 + 存储结构 + 数据运算**

**① 逻辑结构**——数据元素之间的"关系"

把数据元素想象成人，逻辑结构就是描述这些人之间的关系：

### ⏱️ 算法复杂度——衡量算法好坏的标准

**时间复杂度**：算法执行需要多少"步骤"（用大O表示）

**从小到大排序（越小越好）：**

```
O(1) < O(log n) < O(n) < O(n log n) < O(n²) < O(n³) < O(2ⁿ) < O(n!)

常数   对数      线性   线性对数     平方     立方     指数     阶乘
最快                                                         最慢
```

**生动理解各个复杂度：**
- `O(1)`：不管数据多少，直接找到。（比如：直接取数组第5个元素）
- `O(log n)`：每次把范围缩小一半。（比如：二分查找，1000个里找，最多10次）
- `O(n)`：数据多少，做多少次。（比如：找最大值，要看每一个）
- `O(n²)`：两层嵌套循环。（比如：冒泡排序）
- `O(2ⁿ)`：极慢！数据稍大就不可用。

**🔥 如何计算时间复杂度？（常考！）**

**口诀**：找到执行次数最多的语句，用大O表示，忽略常数和低阶项

```c
// 例1：O(n²) —— 二重嵌套循环
for(int i = 0; i < n; i++) {       // 外循环执行n次
    for(int j = 0; j < n; j++) {   // 内循环每次也执行n次
        sum++;                      // 这行执行了 n×n = n² 次
    }
}
// 答案：O(n²)

// 例2：O(log n) —— 每次翻倍
for(int i = 1; i < n; i = i * 2) {
    // i: 1, 2, 4, 8, 16... 直到超过n
    // 需要执行 log₂(n) 次
    printf("%d", i);
}
// 答案：O(log n)

// 例3：O(n log n) —— 典型：归并排序、快速排序（平均）

// 例4：多段代码，取最大的
for(int i = 0; i < n; i++) sum1++;   // O(n)
for(int i = 0; i < n; i++)
    for(int j = 0; j < n; j++) sum2++; // O(n²)
// 整体答案：O(n²)（取最大的）
```

**空间复杂度**：算法需要额外用多少内存

- `O(1)`：只用固定数量的变量，不管数据多大（最好）
- `O(n)`：需要额外开一个和数据等大的数组

## 第2章：线性表——数据的"排队"

### 💡 什么是线性表？

线性表就是数据元素像排队一样，**一个接一个**，有顺序，有前后关系。

特征：
- 除第一个元素外，每个元素都有且仅有一个"前驱"
- 除最后一个元素外，每个元素都有且仅有一个"后继"
- 就像一根项链

### 📋 顺序表（用数组实现）

顺序表就是把数据**按顺序存在连续的内存格子**里，就像停车场——每辆车占一个格，顺序排开。

```
内存地址：  100  104  108  112  116
数据：     [10] [20] [30] [40] [50]
逻辑位置：  第1  第2  第3  第4  第5
```

**优点**：
- 随机访问O(1)——知道第几个，直接算内存地址，瞬间找到
- 不需要额外的指针空间

**缺点**：
- 插入/删除需要移动大量元素O(n)——在中间插一个，后面的全部后移

**插入演示（在位置2插入99）：**
```
插入前：[10][20][30][40][50]
                ↑ 想在这里插入99
步骤1：[10][20][30][40][50][50]  ← 50后移
步骤2：[10][20][30][40][40][50]  ← 40后移
步骤3：[10][20][30][30][40][50]  ← 30后移
步骤4：[10][20][99][30][40][50]  ← 插入99
```
平均需要移动 n/2 个元素，所以是 **O(n)**

---

### 🔗 链表（用指针连接的节点）

链表把每个数据元素放在一个"节点"里，节点包含：
1. **数据域**：存储数据
2. **指针域**：存储下一个节点的地址

```
节点结构：
┌──────┬──────┐
│ data │ next │
│ 数据  │ 指针  │
└──────┴──────┘

链表示意图：
头节点→[10|→]→[20|→]→[30|→]→[40|NULL]
        第1个   第2个   第3个   第4个（末尾指向NULL）
```

> **头节点**：不存数据，只是链表的入口（很多题会用到！）

**① 单链表**

```
NULL ← head
head → [data₁|next₁] → [data₂|next₂] → [data₃|NULL]
```

**② 双向链表**（每个节点有两个指针，可以往前往后走）

```
NULL ← [prev|data₁|next] ↔ [prev|data₂|next] ↔ [prev|data₃|next] → NULL
```

**③ 循环链表**（尾节点的next指向头节点，形成环）

```
head → [A|→] → [B|→] → [C|→]
  ↑___________________________________|
```

---

### ⚡ 顺序表 vs 链表（必考对比！）

```
┌──────────────┬──────────────────────┬──────────────────────┐
│    特性       │       顺序表          │        链表           │
├──────────────┼──────────────────────┼──────────────────────┤
│ 存储方式      │ 连续内存（像停车场）    │ 离散内存（用指针连接） │
├──────────────┼──────────────────────┼──────────────────────┤
│ 随机访问      │ O(1) ✅ 直接计算地址   │ O(n) ❌ 要从头遍历    │
├──────────────┼──────────────────────┼──────────────────────┤
│ 插入/删除     │ O(n) ❌ 要移动元素     │ O(1) ✅ 改指针就行    │
│ （找到位置后） │                      │ （前提是已找到位置）    │
├──────────────┼──────────────────────┼──────────────────────┤
│ 空间利用率    │ 高（无额外指针空间）    │ 低（每个节点有指针域） │
├──────────────┼──────────────────────┼──────────────────────┤
│ 适用场景      │ 查找多，增删少         │ 增删多，查找少         │
└──────────────┴──────────────────────┴──────────────────────┘
```

### 💻 链表操作代码（必考！）

**头插法建立链表（结果是逆序的！）：**

```c
// 想象：每次新同学来了都插到队伍最前面
void CreateList_Head(LinkList &L, int n) {
    L = (LinkList)malloc(sizeof(LNode));  // 创建头节点
    L->next = NULL;                       // 头节点初始没有后继
    
    for(int i = 0; i < n; i++) {
        LNode *p = (LNode*)malloc(sizeof(LNode));  // 新建节点
        scanf("%d", &p->data);           // 输入数据
        p->next = L->next;               // 新节点的next = 原来的第一个节点
        L->next = p;                     // 头节点的next = 新节点（新节点变第一个）
    }
}
// 规律：输入 1,2,3 → 链表是 3→2→1（逆序！）
```

**尾插法建立链表（结果是正序的！）：**

```c
// 想象：每次新同学都站到队尾
void CreateList_Tail(LinkList &L, int n) {
    L = (LinkList)malloc(sizeof(LNode));
    LNode *r = L;  // r指向当前的最后一个节点（初始是头节点）
    
    for(int i = 0; i < n; i++) {
        LNode *p = (LNode*)malloc(sizeof(LNode));
        scanf("%d", &p->data);
        r->next = p;    // 当前尾节点指向新节点
        r = p;          // r更新，指向新的尾节点
    }
    r->next = NULL;     // 最后一个节点的next设为NULL
}
// 规律：输入 1,2,3 → 链表是 1→2→3（正序！）
```

**链表反转（经典考题！）：**

```c
// 方法：用头插法逐个节点插到新链表前面
LinkList Reverse(LinkList L) {
    LNode *p = L->next;  // p指向第一个数据节点
    LNode *q;            // 临时变量保存下一个节点
    L->next = NULL;      // 断开头节点，把旧链表"清空"
    
    while(p) {
        q = p->next;         // 先保存p的下一个
        p->next = L->next;   // p插到头节点后面
        L->next = p;         // 头节点指向p
        p = q;               // p移到下一个待处理节点
    }
    return L;
}
// 原来：1→2→3→4 
// 反转：4→3→2→1
```

## 第3章：栈和队列——特殊的线性表

### 📚 栈（Stack）——"后进先出"（LIFO）

**生活类比**：一摞书，最后放上去的书，最先被拿走

```
            ┌───┐
   push入栈→ │ 4 │ ← 栈顶（top）
            ├───┤
            │ 3 │
            ├───┤
            │ 2 │
            ├───┤
            │ 1 │
            └───┘ ← 栈底（bottom）
   pop出栈←  （从栈顶取）
```

**只能在一端（栈顶）进行操作：**
- `push(x)`：入栈，把x放到栈顶
- `pop()`：出栈，把栈顶元素取出
- `peek()`：查看栈顶元素但不取出

**顺序栈的关键判断：**
```
空栈判断：top == -1
满栈判断：top == MaxSize - 1
入栈操作：top++，然后 stack[top] = x
出栈操作：x = stack[top]，然后 top--
```

**🔥 两栈共享空间（考点！）：**

```
两个栈共用同一片内存，一个从左端开始，一个从右端开始：

左端→ [S1的底] ... [S1的顶] ... 空 ... [S2的顶] ... [S2的底] ←右端
      top1=0                                         top2=MaxSize-1

判断栈满（两栈相遇）：top1 + 1 == top2
```

---

### 🔥 栈的应用（超重要！常出大题）

**应用1：括号匹配**

```
算法思路：
- 遇到左括号（ [ { ）：入栈
- 遇到右括号） ] } ）：取出栈顶，看是否匹配
  - 匹配则继续
  - 不匹配则报错
- 最后栈空则括号全部匹配

例：检验 "( [ { } ] )" 是否合法
扫描到 (   → 栈: [(]
扫描到 [   → 栈: [(, []
扫描到 {   → 栈: [(, [, {]
扫描到 }   → 栈顶是{，匹配，出栈 → 栈: [(, []
扫描到 ]   → 栈顶是[，匹配，出栈 → 栈: [(]
扫描到 )   → 栈顶是(，匹配，出栈 → 栈: []（空）
结果：合法！
```

**应用2：表达式求值（中缀转后缀）——408必考大题！**

```
中缀表达式：我们平时写法，运算符在两个数之间
  如：A + B * C - D

后缀表达式（逆波兰式）：计算机处理方便，运算符在数的后面
  如：A B C * + D -

为什么要转换？因为计算机用栈处理后缀表达式很简单！

中缀 → 后缀 的转换规则：
  ① 从左到右扫描
  ② 遇到数字/字母：直接输出
  ③ 遇到运算符：
     - 如果栈空或栈顶是左括号：直接入栈
     - 如果优先级 > 栈顶运算符：入栈
     - 否则：弹出栈顶，输出，再比较，直到可以入栈
  ④ 遇到左括号(：入栈
  ⑤ 遇到右括号)：弹出并输出，直到弹出左括号（左括号不输出）
  ⑥ 扫描结束：弹出所有剩余运算符

例：A + B * C - D → 后缀
  扫A → 输出: A
  扫+ → 栈空，入栈。栈:[+]  输出: A
  扫B → 输出: A B
  扫* → *优先级>+，入栈。栈:[+,*]  输出: A B
  扫C → 输出: A B C
  扫- → -优先级≤*，弹出*输出; -优先级≤+，弹出+输出; 入栈-
       栈:[-]  输出: A B C * +
  扫D → 输出: A B C * + D
  结束 → 弹出-  输出: A B C * + D -

结果：A B C * + D -  ✓
```

### 🚌 队列（Queue）——"先进先出"（FIFO）

**生活类比**：排队买票，先来先买到

```
入队（enqueue）→ [D][C][B][A] → 出队（dequeue）
                  ↑              ↑
                 队尾           队头
```

**🔑 循环队列（必考！）**

为什么要循环？因为普通队列出队后，前面的空间不能再用（"假溢出"问题）。

循环队列用取模运算让指针"循环"：

```
循环队列示意图（数组大小为7）：

     0    1    2    3    4    5    6
   [   ] [ A ] [ B ] [ C ] [ D ] [   ] [   ]
              ↑                   ↑
           front=1              rear=5
（队列中有元素：A, B, C, D）

出队A后：front=(1+1)%7=2
入队E后：rear=(5+1)%7=6，Q[6]=E
```

**关键公式（必背！）：**
```
判断空队列：front == rear
判断满队列：(rear + 1) % MaxSize == front
           （牺牲一个空间来区分空和满）

队列长度：(rear - front + MaxSize) % MaxSize

入队操作：
  if (队列未满)
      rear = (rear + 1) % MaxSize
      Q[rear] = x

出队操作：
  if (队列非空)
      front = (front + 1) % MaxSize
      x = Q[front]
```

## 第4章：串（字符串）

### 📝 基础概念

```
串：由零个或多个字符组成的有限序列，写作 "hello"

空串：长度为0的串，写作 ""
空格串：含有空格的串，如 " "（长度不为0！）
子串：主串中任意连续字符组成的序列，如 "hel" 是 "hello" 的子串
```

**串的模式匹配**：在主串S中，查找模式串T第一次出现的位置

### 🔥 KMP算法（几乎年年考！）

**为什么需要KMP？**

朴素的暴力匹配（BF算法）：每次失配，主串指针退回，效率低 O(mn)

KMP的聪明之处：**主串指针不回退！**失配时，模式串指针跳到next数组指定的位置

```
KMP时间复杂度：O(m+n)  vs  BF算法：O(mn)
（m=模式串长，n=主串长）
```

**next数组（失配函数）的含义：**

```
next[j] = 当模式串第j个字符匹配失败时，
          模式串应该退回到第next[j]个字符重新尝试匹配
```

**计算next数组（重要！）：**

next[j] 的值 = 模式串前j-1个字符中，**最长的相等前缀和后缀的长度** + 1

```
前缀：从头开始的子串，如 "ab" 的前缀有 {"a", "ab"}（通常不包含整体）
后缀：从尾部开始的子串，如 "ab" 的后缀有 {"b", "ab"}（通常不包含整体）

例：模式串 T = "a b a b c"
         索引  1 2 3 4 5

next[1] = 0  （规定）
next[2] = 1  （规定）
next[3]：看前2个字符 "ab"
   前缀：{a}，后缀：{b}，无相等 → next[3] = 1
next[4]：看前3个字符 "aba"
   前缀：{a, ab}，后缀：{a, ba}，"a"相等，长度1 → next[4] = 1+1 = 2
next[5]：看前4个字符 "abab"
   前缀：{a, ab, aba}，后缀：{b, ab, bab}，"ab"相等，长度2 → next[5] = 2+1 = 3

所以 next = [0, 1, 1, 2, 3]
```

**KMP匹配过程示例：**

```
主串 S = "a b a b c a b a b c d"
模式串T = "a b a b c"
next   =  [0,1,1,2,3]

当T[5]='c' 和 S[某位置] 匹配失败时：
  j = next[5] = 3
  从T[3]='a' 继续比较，主串指针不动！

（比BF算法少了很多无用的比较）
```

## 第5章：树与二叉树——最重要的非线性结构

### 🌳 树的基本概念

**用家谱来理解：**

```
              爷爷（根节点/Root）
            /   |   \
          爸爸  叔叔  姑姑    ← 爷爷的孩子，他们互为兄弟节点
          / \         |
        我  弟弟      表弟    ← 叶节点（没有孩子的节点）

```

**重要术语：**
```
根节点：没有父节点的节点（树的"老祖宗"）
叶节点：没有孩子的节点（树的末端）
节点的度：一个节点拥有的孩子个数（爷爷度=3，爸爸度=2，我度=0）
树的度：所有节点中度的最大值
树的深度（高度）：从根到最远叶节点的路径上节点数
```

**必背公式：**
```
总节点数 = 所有节点的度之和 + 1
原因：每个节点（除根节点外）都是某人的孩子，即占用了一条边
      所有度之和 = 总边数 = 总节点数 - 1（因为根没有父边）
      所以：总节点数 = 所有度之和 + 1
```

---

### 🌲 二叉树——408最重要的数据结构！

**二叉树的特点：**
- 每个节点**最多**有2个孩子（可以0个、1个、2个）
- 孩子有左右之分（左孩子 ≠ 右孩子）

```
        1
       / \
      2   3
     / \   \
    4   5   6

这是一棵二叉树：
节点1有左孩子2和右孩子3
节点2有左孩子4和右孩子5
节点3只有右孩子6
节点4,5,6是叶节点
```

**特殊的二叉树：**

```
满二叉树：每一层都填满，没有"空位"
         若高度为h，则节点数 = 2^h - 1

        1
       / \
      2   3
     /\   /\
    4  5 6  7
    （高度3，节点数=7=2³-1）

完全二叉树：叶节点只在最后两层，且最后一层的叶节点全部靠左
         （满二叉树一定是完全二叉树，反之不一定）

        1
       / \
      2   3
     /\  /
    4  5 6
    （最后层靠左，是完全二叉树）
```

**🎯 二叉树的5个重要性质（必背！）：**

```
性质1：第 i 层最多有 2^(i-1) 个节点（根是第1层）
       第1层最多1个，第2层最多2个，第3层最多4个...

性质2：高度为 h 的二叉树最多有 2^h - 1 个节点（满二叉树时取等）

性质3：★★★最重要★★★
       叶节点数 n₀ = 度为2的节点数 n₂ + 1
       即：n₀ = n₂ + 1
       
       怎么记：树里所有节点的度之和 = 总边数 = n-1（n为总节点数）
       n = n₀ + n₁ + n₂（0度节点 + 1度节点 + 2度节点）
       边数 = n₁×1 + n₂×2 = n-1
       推导：n₀ + n₁ + n₂ - 1 = n₁ + 2n₂
             所以 n₀ = n₂ + 1 ✓

性质4：n个节点的完全二叉树高度 = ⌊log₂n⌋ + 1

性质5：完全二叉树从1开始按层编号，编号为i的节点：
       父节点编号 = ⌊i/2⌋
       左孩子编号 = 2i
       右孩子编号 = 2i+1
       
       这就是为什么数组可以直接存二叉树！
```

---

### 🚶 二叉树的遍历（超级重要！必须手推会）

**遍历 = 按某种顺序，把每个节点访问一次**

```
示例树：
        A
       / \
      B   C
     / \   \
    D   E   F
```

**① 前序遍历（Pre-order）：根 → 左 → 右**
```
结果：A B D E C F

记忆：先访问根，再左子树，再右子树
（根在最"前"面，所以叫前序）
```

**② 中序遍历（In-order）：左 → 根 → 右**
```
结果：D B E A C F

记忆：根在"中"间，先左子树，再根，再右子树
重要：二叉搜索树的中序遍历 = 升序排列！
```

**③ 后序遍历（Post-order）：左 → 右 → 根**
```
结果：D E B F C A

记忆：根在最"后"面，先左子树，再右子树，最后访问根
应用：计算表达式树、删除树（先删子节点再删根）
```

**④ 层序遍历（Level-order）：按层从上到下，每层从左到右**
```
结果：A B C D E F

实现：用队列！
步骤：根节点入队 → 出队访问 → 将其左右孩子入队 → 重复
```

**📌 由两种遍历序列还原二叉树（常考！）：**

> **必须有中序才能唯一确定！**（前序+后序不一定能唯一确定）

```
已知前序：A B D E C F
已知中序：D B E A C F

还原步骤：
1. 前序第一个字母 A 是根节点
2. 在中序中找 A → 左边 {D B E} 是左子树，右边 {C F} 是右子树
3. 左子树：
   前序中 A 后面是 B D E（左子树的前序）
   中序是 D B E（左子树的中序）
   → B 是左子树的根
   → B 在中序中，左边 {D} 是左子树（只有D），右边 {E} 是右子树（只有E）
4. 右子树：
   前序中是 C F（右子树的前序）
   中序是 C F（右子树的中序）
   → C 是右子树的根
   → C 右边只有 F，所以 F 是 C 的右孩子

最终还原的树就是上面那棵示例树！
```

**代码实现（递归遍历）：**
```c
// 前序遍历
void PreOrder(BiTree T) {
    if(T != NULL) {
        printf("%c", T->data);   // 1. 访问根节点
        PreOrder(T->lchild);     // 2. 递归遍历左子树
        PreOrder(T->rchild);     // 3. 递归遍历右子树
    }
    // T==NULL 时直接返回（递归基）
}
// 中序遍历：把上面3句顺序改为 2,1,3
// 后序遍历：把上面3句顺序改为 2,3,1
```

### 🔍 线索二叉树

**问题**：n个节点的二叉树有 **n+1个空指针**（可以证明），浪费空间！

**线索二叉树的思路**：把空指针利用起来，让它们指向遍历的前驱/后继！

```
规则：
- 若节点左孩子为空，左指针指向该节点在某种遍历下的"前驱"
- 若节点右孩子为空，右指针指向该节点在某种遍历下的"后继"
- 用 tag 标记：0 = 是孩子指针，1 = 是线索
```

```
中序线索化后的树（中序：D B E A C F）：

节点D：无左孩子→左线索指向NULL（无前驱），无右孩子→右线索指向B（后继）
节点B：左孩子正常（指D），右孩子正常（指E）
节点E：无左孩子→左线索指向B（前驱），无右孩子→右线索指向A（后继）
...
```

**好处**：可以 O(1) 找到任意节点的前驱/后继，不用每次重新遍历！

### 🌲 二叉搜索树（BST）

**定义（重要！）：**
- 每个节点的值 > 其左子树中所有节点的值
- 每个节点的值 < 其右子树中所有节点的值

```
        5
       / \
      3   7
     / \ / \
    2  4 6  8

中序遍历 → 2 3 4 5 6 7 8（升序！）
这就是BST的关键性质：中序遍历得到有序序列
```

**操作复杂度：**
```
平均情况（树比较平衡）：O(log n)
最坏情况（退化为链表，如依次插入1,2,3,4,5...）：O(n)

    1
     \
      2
       \
        3
         \
          4
（退化链表，查找O(n)）
```

### 🌳 平衡二叉树（AVL树）

**问题**：BST在最坏情况下退化为链表，查找O(n)

**解决**：AVL树保证任意节点的**平衡因子（左右子树高度差）的绝对值 ≤ 1**

```
平衡因子 = 左子树高度 - 右子树高度
|平衡因子| ≤ 1 才是AVL树
```

**四种旋转操作（当插入节点导致失衡时）：**

```
LL型：新节点在失衡节点的左子树的左边 → 右旋一次
      
失衡前：          C(失衡)     旋转后：    B
               /                        / \
              B                        A   C
             /
            A

RR型：新节点在失衡节点的右子树的右边 → 左旋一次（LL的镜像）

LR型：新节点在失衡节点的左子树的右边 → 先左旋再右旋
RL型：新节点在失衡节点的右子树的左边 → 先右旋再左旋
```

**口诀**：
```
"LL→右旋，RR→左旋，LR→先左后右，RL→先右后左"
记忆：L代表Left左，R代表Right右，旋转方向与字母方向相反
```

### 🌲 哈夫曼树（最优二叉树）

**目的**：用于数据压缩，让频繁出现的字符用短编码表示

**带权路径长度（WPL）**：
```
WPL = Σ（叶节点的权值 × 该叶节点到根的路径长度）
```

**构建哈夫曼树的步骤：**
```
给定权值：{2, 3, 4, 7}

步骤1：选最小的两个（2和3），合并成权值5的新节点
       权值集合：{4, 5, 7}，新节点5的左右孩子是2和3

步骤2：选最小的两个（4和5），合并成权值9的新节点
       权值集合：{7, 9}

步骤3：将7和9合并成权值16的根节点

最终树：
          16
         /  \
        7    9
            / \
           4   5
              / \
             2   3

WPL = 7×1 + 4×2 + 2×3 + 3×3 = 7 + 8 + 6 + 9 = 30
```

**哈夫曼编码（前缀码）：**
```
规则：向左走=0，向右走=1（或反过来）

上例中：
7 → 0（路径1步）
4 → 10（路径2步）
2 → 110（路径3步）
3 → 111（路径3步）

特点：任何编码都不是其他编码的前缀（前缀码性质），保证解码无歧义
如果7出现很多，用短编码0；2和3出现少，用长编码110/111，节省空间！
```

## 第6章：图——最复杂的数据结构

### 🕸️ 图的基本概念

```
图 G = (V, E)
V：顶点集合（所有节点）
E：边集合（所有连接关系）

有向图：边有方向（用箭头表示，叫"弧"）
无向图：边无方向

      无向图          有向图
    A ─── B         A ──→ B
    │     │         │
    D ─── C         ↓
                    D
```

**重要术语：**
```
度（无向图）：与节点相连的边数
入度（有向图）：指向该节点的边数
出度（有向图）：从该节点出发的边数

路径：从一个节点到另一个节点经过的边
简单路径：路径中不重复经过同一节点
回路（环）：起点和终点相同的路径

连通图（无向）：任意两点之间都有路径
强连通图（有向）：任意两点之间都有路径（A到B，B到A都有）
```

**完全图（边数最多）：**
```
无向完全图：n个顶点，边数 = n(n-1)/2
有向完全图：n个顶点，边数 = n(n-1)
```

---

### 💾 图的存储方式

**邻接矩阵（适合稠密图）：**
```
用一个 n×n 的矩阵表示
matrix[i][j] = 1 表示 i 和 j 之间有边
matrix[i][j] = 0 表示没有边

    A B C D
A [ 0 1 0 1 ]
B [ 1 0 1 0 ]
C [ 0 1 0 1 ]
D [ 1 0 1 0 ]

优点：O(1)判断两点是否相邻，适合稠密图（边多的图）
缺点：空间O(n²)，稀疏图（边少的图）浪费严重
```

**邻接表（适合稀疏图）：**
```
每个顶点维护一个链表，存储其所有邻居

A → [B] → [D] → NULL
B → [A] → [C] → NULL
C → [B] → [D] → NULL
D → [A] → [C] → NULL

优点：空间O(n+e)，节省空间
缺点：O(degree)才能判断两点是否相邻
```

---

### 🚶 图的遍历

**BFS（广度优先搜索）—— 像水波纹扩散**

```
用队列实现！

从顶点A开始BFS：
            A
           / \
          B   C
         / \
        D   E

步骤：
1. A入队，标记已访问。队列:[A]
2. 出队A，访问A，A的邻居B,C入队。队列:[B,C]
3. 出队B，访问B，B的未访问邻居D,E入队。队列:[C,D,E]
4. 出队C，访问C，C无未访问邻居。队列:[D,E]
5. 出队D，访问D。队列:[E]
6. 出队E，访问E。队列:[]
BFS结果：A B C D E
```

**DFS（深度优先搜索）—— 一条道走到黑**

```
用栈/递归实现！

从顶点A开始DFS（邻居按字母顺序）：
1. 访问A，从A出发，沿B走
2. 访问B，从B出发，沿D走
3. 访问D，D无未访问邻居，回退到B
4. 从B出发，沿E走
5. 访问E，E无未访问邻居，回退到A
6. 从A出发，沿C走
7. 访问C，C无未访问邻居，结束
DFS结果：A B D E C
```

---

### 🌲 最小生成树（MST）

**问题**：n个城市，如何修路让所有城市相连，且总路程最短？

答案就是**最小生成树**：选 n-1 条边，连通所有 n 个节点，且边的权值之和最小

**Prim算法（加点法，适合稠密图）：**
```
思路：从一个点出发，每次加一条"最短的连接已选集合和未选集合的边"

步骤：
1. 任选一个起点放入"已选集合"
2. 找从已选集合出发，到未选集合中权值最小的边
3. 将该边的终点加入已选集合
4. 重复2-3，直到所有节点都在已选集合中
时间复杂度：O(V²)
```

**Kruskal算法（加边法，适合稀疏图）：**
```
思路：将所有边按权值排序，每次取最短的边（如果不会形成环）

步骤：
1. 所有边按权值从小到大排序
2. 依次取边，若该边的两端点不在同一集合（不会形成环）
3. 则加入最小生成树，合并两个集合（用并查集）
4. 重复直到选了 n-1 条边
时间复杂度：O(E log E)
```

---

### 🛣️ 最短路径

**Dijkstra算法（单源最短路径，不能有负权边）：**

```
思路：贪心算法，每次选距离源点最近的未访问节点

步骤：
1. 初始化：dist[源点]=0，其他节点dist=∞
2. 选未访问中dist最小的节点v
3. 更新v的所有邻居：如果 dist[v]+边权 < dist[邻居]，则更新dist[邻居]
4. 标记v已访问
5. 重复2-4

例：求A到所有点的最短路
        2
    A ──── B
    │      │
  3 │    1 │
    │      │
    C ──── D
        4

初始：dist[A]=0, dist[B]=∞, dist[C]=∞, dist[D]=∞
取A（dist最小）：更新B=2, C=3
取B（dist=2）：更新D=2+1=3
取C（dist=3）：D是否更新？3+4=7>3，不更新
取D（dist=3）：已更新完毕
结果：A→A=0, A→B=2, A→C=3, A→D=3
```

**Floyd算法（多源最短路径，可以有负权边）：**
```
思路：动态规划，枚举所有可能的中间节点

核心代码：
for k in 1..n:           ← 枚举中间节点k
  for i in 1..n:
    for j in 1..n:
      dist[i][j] = min(dist[i][j], dist[i][k] + dist[k][j])
      （经过k中转是否更短？）

时间复杂度：O(n³)，适合节点数少的稠密图
```

### 📋 拓扑排序（有向无环图DAG）

**应用场景**：课程安排（数学必须在微积分之前学）、工序安排

```
规则：
1. 选入度为0的节点（没有先修要求），放入结果序列
2. 删除该节点及其所有出边（这些课程的先修条件已满足）
3. 重复，直到所有节点被选出（成功）或没有入度为0的节点（有环，失败）

例：
A → C → E
B → C
B → D → E

入度：A=0, B=0, C=2, D=1, E=2

取A（入度0）：删A的边，C的入度变1
取B（入度0）：删B的边，C的入度变0，D的入度变0
取C（入度0）：删C的边，E的入度变1
取D（入度0）：删D的边，E的入度变0
取E（入度0）：完成！
结果：A B C D E（或 B A C D E，可能有多种）
```

## 第7章：查找

### 🔍 各种查找方法

**① 顺序查找（线性查找）**
```
方法：从头到尾一个个看
时间复杂度：O(n)
平均比较次数：(n+1)/2
适用：无序或有序表都行，但效率低
```

**② 折半查找（二分查找）**
```
前提：顺序存储 + 有序序列
方法：每次与中间元素比较，不行就去左半或右半
时间复杂度：O(log n)

代码：
int BinarySearch(int A[], int n, int key) {
    int low = 0, high = n-1;
    while(low <= high) {
        int mid = (low + high) / 2;
        if(A[mid] == key) return mid;     // 找到
        else if(A[mid] > key) high = mid-1;  // 去左半
        else low = mid+1;                 // 去右半
    }
    return -1;  // 没找到
}

例：在 [1,3,5,7,9,11,13] 中找 7
    low=0, high=6, mid=3 → A[3]=7 → 找到！
```

**③ B树（数据库索引）**

```
m阶B树：每个节点最多有m个孩子，用于减少磁盘访问次数

核心性质：
1. 每个节点最多m个孩子
2. 非根非叶节点至少⌈m/2⌉个孩子（减少层数）
3. 根节点至少2个孩子（除非是叶节点）
4. 所有叶节点在同一层（平衡！）
5. 节点中关键字数 = 孩子数 - 1

用途：数据库、文件系统多级索引
优点：一次读入整个节点，减少磁盘I/O
```

**B+树 vs B树（重要对比）：**
```
┌────────────┬───────────────────┬───────────────────────┐
│  特点       │      B树           │      B+树             │
├────────────┼───────────────────┼───────────────────────┤
│ 数据存放    │ 所有节点都有数据    │ 只有叶节点有数据        │
├────────────┼───────────────────┼───────────────────────┤
│ 叶节点      │ 独立              │ 用链表串联（有序！）     │
├────────────┼───────────────────┼───────────────────────┤
│ 查找次数    │ 不稳定（可能在     │ 稳定，总要走到叶节点    │
│            │ 中间节点就找到）   │                        │
├────────────┼───────────────────┼───────────────────────┤
│ 范围查询    │ 慢（要回溯）       │ 快！沿叶链表扫描即可    │
└────────────┴───────────────────┴───────────────────────┘
MySQL的InnoDB用的就是B+树！
```

**④ 哈希查找（散列查找）**

```
思路：根据关键字，用哈希函数直接计算出存储地址，O(1)查找！

哈希函数：H(key) = key mod m（m一般取质数）
例：m=7，存储 {15, 22, 8, 3}
H(15) = 15 mod 7 = 1 → 放在位置1
H(22) = 22 mod 7 = 1 → 冲突！位置1已有15
H(8)  = 8 mod 7 = 1  → 又冲突！

冲突（不同key映射到同一位置）是哈希的最大问题！
```

**冲突解决方法：**
```
方法1：开放定址法（冲突了就找下一个空位）
  ① 线性探测：H(key) 冲突 → 试 H(key)+1 → H(key)+2 → ...
  ② 二次探测：H(key)+1² → H(key)-1² → H(key)+2² → H(key)-2² → ...

方法2：链地址法（哈希值相同的元素用链表连接）
  位置1: [15] → [22] → [8] → NULL
  最常用！
  
装填因子 α = 元素个数 / 哈希表长度
α越小，冲突越少，查找效率越高
```

### ⚡ 快速排序（最常考！一定要会手推）

**思路**：选一个"基准（pivot）"，把小的放左边，大的放右边，然后递归

```c
// 一趟划分：选第一个为pivot，一次归位
int Partition(int A[], int low, int high) {
    int pivot = A[low];    // 选基准
    while(low < high) {
        // 从右向左找小于pivot的
        while(low < high && A[high] >= pivot) high--;
        A[low] = A[high];  // 移到左边
        // 从左向右找大于pivot的
        while(low < high && A[low] <= pivot) low++;
        A[high] = A[low];  // 移到右边
    }
    A[low] = pivot;        // pivot归位
    return low;            // 返回pivot的位置
}

void QuickSort(int A[], int low, int high) {
    if(low < high) {
        int p = Partition(A, low, high);  // pivot归位，返回其位置
        QuickSort(A, low, p-1);   // 递归排左半部分
        QuickSort(A, p+1, high);  // 递归排右半部分
    }
}
```

**手推示例：**
```
初始：[3, 1, 4, 1, 5, 9, 2, 6]
       ↑ pivot=3，low=0，high=7

从右找小于3的：A[6]=2 < 3，A[7]=6≥3，停在index=6
移动：[2, 1, 4, 1, 5, 9, _, 6]（_表示pivot的空位）

从左找大于3的：A[0]=2（已换），A[1]=1≤3，A[2]=4>3，停在index=2
移动：[2, 1, _, 1, 5, 9, 4, 6]

从右找小于3的：A[5]=9≥3，A[4]=5≥3，A[3]=1<3，停在index=3
移动：[2, 1, 1, _, 5, 9, 4, 6]

从左找大于3的：A[3]=（已换），low=high=3，退出循环
放入pivot：[2, 1, 1, 3, 5, 9, 4, 6]
                   ↑ pivot=3归位！左边全≤3，右边全≥3

递归排左[2,1,1]和右[5,9,4,6]
```

**快排最坏情况**：已排好序的数组（每次选最大或最小为pivot），退化为O(n²)

### 🔺 堆排序

**堆的概念：**
```
大根堆：任意节点的值 ≥ 其子节点的值（堆顶是最大值）
小根堆：任意节点的值 ≤ 其子节点的值（堆顶是最小值）

堆用完全二叉树表示，存在数组里（利用完全二叉树性质5）
```

**堆排序步骤（升序用大根堆）：**
```
步骤1：建堆
  从最后一个非叶节点（index = n/2）开始，向前逐个"下沉"（sift down）
  
步骤2-n：反复操作
  ① 堆顶（最大值）与最后一个元素交换
  ② 堆的大小减1（最大值已排好位置）
  ③ 对新的堆顶进行下沉操作（恢复堆性质）
  ④ 重复直到堆只剩1个元素

例：初始 [4, 10, 3, 5, 1]
建堆后：[10, 5, 3, 4, 1]（大根堆）

第1轮：交换10和1 → [1, 5, 3, 4, | 10]（10排好）
       下沉1 → [5, 4, 3, 1, | 10]

第2轮：交换5和1 → [1, 4, 3, | 5, 10]
       下沉1 → [4, 1, 3, | 5, 10]

...最终得到升序数组
```

**思路（分治法）：**
```
1. 把数组一分为二
2. 分别对左半和右半排序（递归）
3. 把两个有序的半段合并成一个有序数组

例：[3, 1, 4, 1, 5, 9, 2, 6]

分割：[3,1,4,1] 和 [5,9,2,6]
递归排序：[1,1,3,4] 和 [2,5,6,9]

[1,1,3,4] + [2,5,6,9]

 1 < 2 → 取左边1

  1 < 2 → 取左边1

    3 > 2 → 取右边2
    ↑          ↑
    3 < 5 → 取左边3
    ...
```

### 🔢 数制转换

**十进制 → 二进制：**
```
整数部分：除以2，取余数（从下往上读）
小数部分：乘以2，取整数（从上往下读）

例：13.625 → 二进制

整数13：
13 ÷ 2 = 6 余 1  ↑ 读的方向
 6 ÷ 2 = 3 余 0  |
 3 ÷ 2 = 1 余 1  |
 1 ÷ 2 = 0 余 1  ↑ 这个先读
结果：1101₂

小数0.625：
0.625 × 2 = 1.25 → 取1  ↓ 读的方向
0.25  × 2 = 0.5  → 取0
0.5   × 2 = 1.0  → 取1（结束）
结果：.101₂

所以：13.625₁₀ = 1101.101₂
```

**二进制 ↔ 十六进制（4位一组）：**
```
1010 0011₂ = A3₁₆
（1010=10=A，0011=3）

十六进制数字：0-9，A=10，B=11，C=12，D=13，E=14，F=15
```

### 📊 带符号数的表示——原码、反码、补码

**为什么需要补码？**

> 因为我们想让计算机只用加法就能做减法！

```
原码：最直观的表示
  正数：最高位(符号位)为0，其余是数的二进制
  负数：最高位为1，其余是数的绝对值的二进制
  
  +7 的8位原码：0000 0111
  -7 的8位原码：1000 0111

反码：
  正数：同原码
  负数：符号位不变，其他位取反（0变1，1变0）
  
  -7 的8位反码：1111 1000（原码除符号位全取反）

补码：（最重要！计算机实际使用的！）
  正数：同原码
  负数：反码 + 1
  
  -7 的8位补码：1111 1000 + 1 = 1111 1001

移码：补码的符号位取反（主要用于浮点数的阶码）
  -7 的8位移码：0111 1001
```

**补码的神奇之处：**
```
+7  的补码：0000 0111
-7  的补码：1111 1001

相加：0000 0111
    +1111 1001
    =10000 0000 → 最高位溢出丢弃 → 0000 0000 = 0

7 + (-7) = 0 ✓  用加法做了减法！

补码特殊值：
-128 的8位补码：1000 0000（没有对应的原码！）
+127 的8位补码：0111 1111
8位补码范围：-128 ~ +127
```

**补码转原码（两种方法）：**
```
方法1：再次取补（对补码取反码，再+1）
方法2：符号位不变，其余取反后+1

例：1111 1001（补码，符号位1表示负数）
方法1：取反→1000 0110，+1→1000 0111，即-7
方法2：符号位不变1，其余1111 001取反→0000 110，+1→0000 111
       →1000 0111，即-7
```

### 🌊 IEEE 754浮点数格式（必考大题！）

**为什么需要浮点数？**

整数格式无法表示 3.14、0.0001这样的小数

**IEEE 754标准格式：**

```
单精度（32位float）：
[符号位S(1)] [阶码E(8)] [尾数M(23)]
   1位           8位          23位

双精度（64位double）：
[符号位S(1)] [阶码E(11)] [尾数M(52)]
   1位           11位         52位
```

**计算公式：**
```
数值 = (-1)^S × 1.M × 2^(E-127)
                      ↑
                 偏置值bias=127（单精度）

注意：尾数的最高位隐含了一个1（规格化数），所以实际精度是1+23=24位

例：将 0 10000001 10000000000000000000000 还原
    S=0（正数）
    E=10000001₂=129（十进制），指数=129-127=2
    M=10000000000000000000000，实际尾数=1.1（二进制）= 1.5
    数值 = +1.5 × 2² = 6.0
```

**特殊值：**
```
┌──────────┬────────┬──────────┬────────────┐
│  情况    │  阶码E │  尾数M   │   含义     │
├──────────┼────────┼──────────┼────────────┤
│ 规格化数  │ 1~254  │  任意    │ 正常浮点数  │
├──────────┼────────┼──────────┼────────────┤
│ 非规格化  │   0    │  非0     │ 极小的数   │
├──────────┼────────┼──────────┼────────────┤
│ 零       │   0    │   0      │  ±0       │
├──────────┼────────┼──────────┼────────────┤
│ 无穷大   │  255   │   0      │  ±∞       │
├──────────┼────────┼──────────┼────────────┤

└──────────┴────────┴──────────┴────────────┘
```

### ✅ 校验码

**奇偶校验码（最简单）：**
```
偶校验：保证所有位（含校验位）中1的个数为偶数
奇校验：保证所有位（含校验位）中1的个数为奇数

例：数据10110100
1的个数=4（偶数）
偶校验位=0（使总数仍为偶数）
奇校验位=1（使总数变为奇数）

缺点：只能检测奇数位的错误，不能检测偶数位的错误，也不能纠错
```

**海明码（Hamming Code，可纠1位错误）：**
```
思路：添加多个校验位，每个校验位负责检验一组特定的数据位

校验位位置：1, 2, 4, 8, 16...（2的幂次位）
数据位放在其他位置

若数据位有n位，需要k位校验位满足：2^k ≥ n + k + 1

检错纠错过程：
每个校验位检查一组数据，如果某位出错，
将所有出错的校验位的编号相加，和就是出错位的编号！
```

**CRC循环冗余校验（检错强，常用于网络传输）：**
```
思路：
1. 原始数据左移k位（k=生成多项式的次数）
2. 用生成多项式做"模2除法"（用XOR代替减法）
3. 余数就是校验码（FCS），附在数据后面发送

接收方：用相同生成多项式对收到的数据做模2除法
        余数为0：无错；非0：有错

例：数据1101，生成多项式 G=101（次数k=2）
  1101 左移2位 → 110100
  110100 ÷ 101（模2除法）：
      110100
   ÷   101
   = 1 余 110100 XOR 101000 = 011100
   = 11 余 011100 XOR 011000 = 000100
   = 110 余 000100 XOR 000000（101比100大，商0）
   余数 = 100
  
  发送：110100 + 100 = 110100100
```

位扩展：增加数据位宽（多芯片并联）
  例：用4片 1K×4bit 的芯片组成 1K×16bit 的存储器
  方法：4片并联，地址线共享，数据线各提供4位，共16位

字扩展：增加地址数量（多芯片串联）
  例：用4片 1K×8bit 的芯片组成 4K×8bit 的存储器
  方法：用地址线的高2位选片（00选第1片，01选第2片...）

综合扩展：
  例：用 256K×8bit 的芯片，组成 1M×32bit 的存储器
  ① 位扩展：32/8=4片并联（凑32位）
  ② 字扩展：1M/256K=4组串联（凑1M地址）
  ③ 总片数 = 4 × 4 = 16片
```

**地址线和数据线的关系：**
```
存储容量 = 2^(地址线根数) × 数据线根数 / 8  （单位：字节）

例：20根地址线，8根数据线
  容量 = 2^20 × 8bit = 1M × 8bit = 1MB
  
例：16根地址线，32根数据线
  容量 = 2^16 × 32bit = 64K × 4字节 = 256KB
```

### ⚡ Cache高速缓存——超级重要！

**为什么需要Cache？**

CPU速度：GHz级（1ns）；内存速度：100ns级；速度差100倍！

Cache就像CPU的"草稿纸"，把最近常用的数据放在旁边，减少等待时间。

**局部性原理（Cache存在的理论基础）：**
```
时间局部性：刚访问过的数据，很快还会再次访问
  （例：循环体中的变量，每次循环都要访问）

空间局部性：访问了某个地址，其附近的地址也很可能被访问
  （例：数组的连续访问，for(i=0;i<n;i++) a[i]）
```

**关键指标：**
```
命中率 h = Cache命中次数 / 总访问次数（越高越好，好的Cache h>95%）

平均访问时间公式（两种情况）：

① 同时查找（先查Cache，如果未命中同时查主存）：
  t = h × t_c + (1-h) × t_m
  （命中时花Cache时间，未命中时花主存时间）

② 先查Cache，未命中再查主存（串行查找）：
  t = h × t_c + (1-h) × (t_c + t_m)
  （未命中时花了Cache时间+主存时间）
```

**🔥 Cache映射方式（必考！）：**

> 主存中的数据怎么放到Cache的哪一行？

```
主存被分成很多"块"（block），Cache也被分成很多"行"（line）
块大小 = 行大小（通常为64字节）

方式1：直接映射（Direct Mapping）
  规则：主存块号 mod Cache总行数 = 放在哪行
  
  例：Cache有8行，主存块0→行0，块1→行1，...，块8→行0，块9→行1...
  
       主存        Cache
      块0    →    行0
      块1    →    行1
      ...         ...
      块7    →    行7
      块8    →    行0  （和块0竞争！）
      块9    →    行1  （和块1竞争！）

  优点：实现简单，硬件便宜
  缺点：命中率低（如果程序频繁访问块0和块8，它们互相驱逐，Cache几乎没用）

方式2：全相联映射（Fully Associative）
  规则：主存块可以放到Cache的任意行
  
  优点：命中率高（最灵活，任何块都不互相干扰）
  缺点：需要比较所有行的标记，硬件复杂，成本高

方式3：组相联映射（Set Associative）★★★最常用！
  规则：Cache分成若干"组"，每组k行
        主存块 → 按直接映射找到对应的"组"
        在该组内 → 按全相联映射放到任意行
  
  例：4路组相联（每组4行）：
       组0：行0,1,2,3
       组1：行4,5,6,7
       ...
       主存块号 mod 组数 = 放在哪个组，组内随意放
  
  k=1：退化为直接映射
  k=所有行：退化为全相联
  实际常用：4路、8路组相联
```

**Cache替换算法（Cache满了时，替换哪一行）：**
```
LRU（最近最少使用）：替换最久没被访问的行（最常用！命中率最高）
FIFO（先进先出）：替换最早进入Cache的行（简单但可能抖动）
LFU（最少使用频次）：替换使用次数最少的行（需要计数器，开销大）
随机替换：随机选一行替换（最简单，平均性能不差）
```

**Cache写策略：**
```
写命中时（要修改的数据在Cache中）：
  写回法（Write Back）：只改Cache，等该行被替换时再写回主存
    优点：速度快（减少主存写次数）
    缺点：某时刻主存数据可能不是最新的（不一致问题）
  
  全写法（Write Through）：同时写Cache和主存
    优点：主存始终是最新的（一致性好）
    缺点：每次写都要访问主存，速度较慢

写未命中时（要修改的数据不在Cache中）：
  写分配法：把该主存块先装入Cache，然后修改Cache
  非写分配法：直接修改主存，不装入Cache
```

### 🌐 虚拟内存

**为什么需要虚拟内存？**

> 内存（物理内存）不够用！程序希望使用比实际内存更大的地址空间。

**核心思想：**
- 给每个程序一个大的"虚拟地址空间"（比如4GB）
- 实际上只把当前用到的部分装入物理内存
- 暂时不用的部分放在硬盘上（磁盘的一块区域叫"交换区"）

```
虚拟地址空间（4GB，程序以为自己有这么多）
     ↕  虚实映射（页表）
物理内存（如1GB，实际只有这么多）
     ↕  缺页时从磁盘调入
磁盘交换区（存放暂时不用的页）
```

**分页虚拟内存：**
```
将虚拟地址空间和物理内存都划分为固定大小的"页"
虚拟页（Virtual Page）→ 物理帧（Physical Frame）

虚拟地址的结构：
[虚拟页号VPN | 页内偏移offset]
      ↓ 查页表
[物理帧号PFN | 页内偏移offset]
      = 物理地址
      
页大小通常4KB，所以页内偏移12位（2^12=4096=4KB）
```

**地址翻译过程（有TLB快表加速）：**
```
1. 取出虚拟地址的VPN（页号部分）
2. 查TLB（快表，Cache化的页表）
   - 命中：直接得到PFN，组合出物理地址
   - 未命中：查主存中的页表
     - 页在内存：得到PFN，更新TLB
     - 页不在内存：缺页中断！从磁盘调入，然后重新访问

TLB命中：只需1次内存访问（取数据）
TLB未命中，页在内存：2次内存访问（查页表+取数据）
缺页中断：几万次内存访问时间（磁盘读取）
```

**页面置换算法（缺页时，换出哪个页？）：**
```
OPT（最优）：换出未来最久不会被使用的页（理论最优，无法实际实现）
LRU（最近最少使用）：换出最久没被使用的页（实际效果好）
FIFO（先进先出）：换出最早进入内存的页
Clock：FIFO的改进版，有使用位，被访问时设为1，换出时如果是0才换出

★ Belady异常（重要考点！）：
  FIFO算法中，分配给进程的物理帧增多，缺页次数反而可能增多！
  LRU和OPT不会发生Belady异常。
```

## 第4章：指令系统

### 📋 指令格式

**一条指令 = 操作码 + 地址码**

```
[操作码 | 地址码1 | 地址码2 | 地址码3]

操作码：告诉CPU做什么（加、减、移动、跳转...）
地址码：告诉CPU操作哪些数据（在哪取数据，结果放哪）

三地址指令：结果放地址1，地址2和地址3是两个操作数
二地址指令：地址1既是操作数又是结果存放处
单地址指令：只有一个地址码
零地址指令：操作数隐含（如Pop操作，操作数隐含在栈顶）
```

**扩展操作码技术：**
```
问题：固定长度指令中，如果操作码短，则指令种类少
解决：让不同长度的指令共存！短操作码处理常见指令，长操作码处理更多指令

例：16位指令，可以这样分配：
操作码4位（1111除外）：15种三地址指令
操作码8位（1111xxxx）：256种二地址指令
操作码12位（11111111xxxx）：更多单地址指令
```

### 🔄 寻址方式（重要！）

> 寻址方式 = 如何找到操作数

```
┌───────────────┬──────────────────────┬──────────────┬─────────────┐
│  寻址方式      │  有效地址（EA）        │  优点         │  缺点       │
├───────────────┼──────────────────────┼──────────────┼─────────────┤
│ 立即寻址       │ 操作数就在指令中       │ 最快，不访存   │ 数值范围小  │
│（如：MOV R,5）│                      │              │             │
├───────────────┼──────────────────────┼──────────────┼─────────────┤
│ 直接寻址       │ EA = A（A是指令中的地址）│ 简单          │ 地址范围受限│
├───────────────┼──────────────────────┼──────────────┼─────────────┤
│ 间接寻址       │ EA = M[A]（A处存的是地址）│ 地址范围大   │ 多一次访存  │
├───────────────┼──────────────────────┼──────────────┼─────────────┤
│ 寄存器寻址     │ EA = 寄存器内容        │ 最快！不访存   │ 寄存器数量少│
├───────────────┼──────────────────────┼──────────────┼─────────────┤
│ 寄存器间接     │ EA = M[寄存器内容]     │ 较快          │ -           │
├───────────────┼──────────────────────┼──────────────┼─────────────┤
│ 基址寻址       │ EA = 基址寄存器 + 偏移 │ 程序重定位     │ -           │
├───────────────┼──────────────────────┼──────────────┼─────────────┤
│ 变址寻址       │ EA = 变址寄存器 + 偏移 │ 数组访问很方便 │ -           │
├───────────────┼──────────────────────┼──────────────┼─────────────┤
│ 相对寻址       │ EA = PC + 偏移         │ 指令跳转       │ -           │
└───────────────┴──────────────────────┴──────────────┴─────────────┘
```

## 第5章：中央处理器（CPU）

### 🧠 CPU的内部结构

```
CPU内部组成：
┌─────────────────────────────────────────────────┐
│                     CPU                         │
│  ┌─────────────────────┐  ┌──────────────────┐  │
│  │    ALU（算术逻辑单元）  │  │   寄存器组        │  │
│  │  负责：加减乘除、        │  │ PC: 程序计数器   │  │
│  │  逻辑运算、移位          │  │ IR: 指令寄存器   │  │
│  └─────────────────────┘  │ MAR: 存储器地址   │  │
│                            │ MDR: 存储器数据   │  │
│  ┌─────────────────────┐  │ PSW: 程序状态字   │  │
│  │    CU（控制器）       │  └──────────────────┘  │
│  │  负责：取指、译码、      │                       │
│  │  产生控制信号          │                       │
│  └─────────────────────┘                       │
└─────────────────────────────────────────────────┘
```

**重要寄存器详解：**
```
PC（Program Counter，程序计数器）：
  存储"下一条要执行的指令的地址"
  每次取指后自动+1（指向下一条）
  跳转指令会修改PC的值

IR（Instruction Register，指令寄存器）：
  存储"当前正在执行的指令"

MAR（Memory Address Register，存储器地址寄存器）：
  CPU要访问内存时，先把地址放入MAR

MDR（Memory Data Register，存储器数据寄存器）：
  CPU从内存读出或写入内存的数据，先存在MDR
  
PSW（Program Status Word，程序状态字寄存器）：
  存放条件码：溢出标志(OF)、进位标志(CF)、负数标志(SF)、零标志(ZF)
```

---

### 🔄 指令的执行过程（必考！）

**一个完整的指令周期 = 取指 + （间址） + 执行 + （中断）**

```
取指周期（每条指令必须有）：
  步骤1：PC → MAR（把下一条指令的地址送到MAR）
  步骤2：主存 → MDR（读出MAR指向的内容到MDR）
  步骤3：MDR → IR（把读出的指令存入IR）
  步骤4：PC + 1 → PC（PC更新，指向下一条指令）
  步骤5：OP(IR) → CU（把指令的操作码送控制器，开始译码）

间址周期（只有使用间接寻址时才有）：
  步骤1：Ad(IR) → MAR（地址码送MAR）
  步骤2：主存 → MDR（读出地址）
  步骤3：MDR → MAR（把读出的真实地址送MAR）
  
执行周期（每条指令必须有）：
  根据操作码，执行具体操作（计算、读写内存、跳转等）

中断周期（有中断请求时才有）：
  保存断点（把PC存入特定地址），跳到中断服务程序
```

### ⚡ 流水线技术（重要！计算题！）

**为什么需要流水线？**

```
不用流水线（串行执行）：
指令1：[取指][译码][执行]
指令2：               [取指][译码][执行]
指令3：                             [取指][译码][执行]
总时间 = 3 × 3t = 9t（每步耗时t）

流水线（并行执行）：
指令1：[取指][译码][执行]
指令2：      [取指][译码][执行]
指令3：            [取指][译码][执行]
总时间 = 3t + 2t = 5t（节省了很多！）

流水线就像工厂的流水线，不同工序同时进行！
```

**流水线的性能指标：**
```
吞吐率（TP）= 完成任务数 / 总时间
           = n / [(k + n - 1) × Δt]
（n=指令数，k=流水线段数，Δt=每段耗时）

加速比 = 不用流水线时间 / 用流水线时间
       = (n × k × Δt) / [(k + n - 1) × Δt]
       = nk / (k + n - 1)

当n→∞（大量指令）时，最大加速比 ≈ k（理论极限）

例：3段流水线（取指、译码、执行），各段时间均为1ns，执行100条指令
  用流水线：(3 + 100 - 1) × 1ns = 102ns
  不用流水线：100 × 3ns = 300ns
  加速比 = 300/102 ≈ 2.94（接近3，即流水线段数）
```

**流水线冒险（Hazard）：**
```
1. 结构冒险（资源冲突）：
   两条指令同时需要同一个硬件资源（如同时要访问内存）
   解决：增加资源（如指令缓存和数据缓存分开）；或插入"气泡"（等待）

2. 数据冒险（数据相关）：
   后面的指令需要前面指令还没计算出的结果
   
   最常见（RAW，先写后读）：
   ADD R1, R2, R3   # R1 = R2 + R3
   SUB R4, R1, R5   # R4 = R1 - R5，但R1还没算好！
   
   解决：
   ① 数据前递（旁路技术）：把ALU结果直接传给下一条，不等写回寄存器
   ② 流水线停顿（插入气泡等待）

3. 控制冒险（分支跳转）：
   遇到 if/jump 指令时，不知道下一条该取哪条指令
   解决：分支预测（预测是否跳转，猜错了再清空流水线）
```

### 🔀 中断系统

**什么是中断？**

> 就像你正在专心做作业，突然手机响了（外部事件发生），
> 你放下笔去接电话（中断处理），接完电话再继续做作业（返回断点）

```
中断的分类：
硬件中断（外中断）：来自CPU外部
  ① 可屏蔽中断（INTR）：如键盘输入、磁盘完成
  ② 不可屏蔽中断（NMI）：如电源故障（必须立即处理）

软件中断（内中断/陷阱）：由程序执行产生
  ① 异常：程序执行出错，如除零、页缺失
  ② 系统调用：程序请求操作系统服务（用软中断指令触发）
```

**中断处理过程：**
```
1. 中断请求：I/O设备等发出中断信号
2. 中断响应：CPU在一条指令执行完后，检查是否有中断
3. 关中断：避免中断处理被打断
4. 保存断点：把PC（返回地址）和PSW保存到特定地址
5. 识别中断源：判断是哪种中断
6. 保存现场：把相关寄存器的值压栈
7. 执行中断服务程序：处理实际的事件
8. 恢复现场：从栈中恢复寄存器
9. 开中断
10. 返回断点：从特定地址取出保存的PC，继续原来的程序
```

## 第6章：总线系统

### 🚌 总线的基本概念

```
总线（Bus）：计算机各部件之间共享的传输通道

按功能分3种：
┌────────────────┬──────────────────────────────────────┐
│  数据总线（DB） │ 传输数据，双向，宽度=机器字长（如64位）│
├────────────────┼──────────────────────────────────────┤
│  地址总线（AB） │ 传输地址，单向，位数决定寻址空间大小    │
│               │ 如32位地址总线→可寻址2^32=4GB          │
├────────────────┼──────────────────────────────────────┤
│  控制总线（CB） │ 传输读写控制、中断等控制信号            │
└────────────────┴──────────────────────────────────────┘

总线带宽 = 总线宽度 × 总线频率
例：64位总线，100MHz
带宽 = 64bit × 100M次/秒 = 6400Mbps = 800MB/s
```

**总线仲裁（多个设备同时想用总线时，谁优先？）：**
```
集中式仲裁：有专门的仲裁器
  ① 链式查询：总线请求信号串行传递，靠近仲裁器的优先级高
     特点：优先级固定，离仲裁器近的设备一直优先
  ② 计数器定时查询：仲裁器用计数器依次询问
     特点：可以设置优先级，也可以轮询（公平）
  ③ 独立请求：每个设备有独立的请求线
     特点：速度最快，但线多

分布式仲裁：各设备自行竞争，无专用仲裁器
  （如以太网的CSMA/CD）
```

## 第7章：输入输出系统（I/O）

### 🖥️ I/O控制方式的演进

**随着技术发展，CPU参与度越来越低，效率越来越高：**

```
方式1：程序查询方式（轮询）
  CPU不停地问设备："你好了吗？好了吗？好了吗？"
  一直等到设备准备好才继续
  
  CPU利用率：极低（一直在"等"）
  适用：简单的单任务系统

方式2：中断方式
  CPU发出I/O请求后，继续做其他事
  设备准备好了主动通知CPU（发中断）
  
  CPU利用率：中等（可以做别的事，但中断处理本身也有开销）
  适用：低速I/O设备（如键盘）

方式3：DMA方式（直接内存存取）★★★
  CPU只需给DMA控制器下达命令（起始地址、数量、方向）
  DMA控制器自行完成内存和I/O设备之间的数据传输
  传输完成后，DMA发中断通知CPU
  
  传输过程中CPU完全不参与，只是在传输期间"偷走"一些总线周期
  CPU利用率：很高
  适用：高速批量传输（如磁盘读写、网卡）

方式4：通道方式
  通道是专用的I/O处理器，有自己的指令集（通道程序）
  CPU启动通道，通道独立完成整个I/O操作
  CPU利用率：最高
  适用：大型主机
```

```
对比总结：
CPU参与度：程序查询 > 中断 > DMA > 通道
传输效率：程序查询 < 中断 < DMA < 通道
```

## 第1章：操作系统概述

### 🎯 什么是操作系统？

```
操作系统（OS）= 管理计算机硬件资源，提供用户接口的系统软件

                用户
                 ↕
            应用程序（QQ, Word...）
                 ↕

1. 处理机管理（CPU管理）

3. 文件管理

4. 设备管理（I/O管理）

虚拟（Virtual）：把一个物理资源虚拟成多个逻辑资源
  如：一个CPU虚拟成多个（多道程序）
  如：少量内存虚拟成大量（虚拟内存）

异步（Asynchronism）：进程的执行是走走停停的，速度不可预知
```

## 第2章：进程与线程

### 🔄 进程的概念

**进程 vs 程序的区别：**
```
程序：硬盘上的可执行文件，静态的（如Word.exe文件）
进程：程序的一次执行过程，动态的（如打开一个Word窗口）

进程是资源分配的基本单位
```

**PCB（进程控制块）—— 进程的"身份证"：**
```
操作系统通过PCB管理每个进程

PCB包含：
- PID（进程标识符）：进程的唯一ID
- 进程状态：就绪/运行/阻塞
- PC（程序计数器）：下一条要执行的指令地址
- 寄存器集合：进程切换时保存现场
- 内存信息：页表基址等
- I/O状态：打开的文件列表
- 优先级等调度信息
```

### 🔄 进程的三种状态（必背！）

```
三态模型：

        调度（分配CPU）
    就绪 ─────────────→ 运行
（Ready）                 │（Running）
    ↑   ←─────────────── │
    │    时间片到/被抢占  │
    │                    │ 等待I/O/资源
    │    I/O完成/        ↓
    │    资源到位        阻塞
    └──────────────── （Blocked）

注意：
阻塞→就绪（I/O完成，不能直接变为运行，要经过调度）
运行→就绪（被抢占，还是就绪态，随时可以再次被调度）
```

**五态模型（加上新建和终止）：**
```
新建 → 就绪 ↔ 运行 → 终止
               ↕
             阻塞

进程的生命周期：
创建（fork/exec） → 就绪队列等待 → 被调度运行 → 
等待资源而阻塞 → 资源到位回到就绪 → 最终终止
```

线程：CPU调度的基本单位（共享进程内存，开销小）

一个进程中可以有多个线程
同一进程的线程共享：代码段、数据段、打开的文件
每个线程私有：PC（程序计数器）、寄存器值、栈

生活类比：
进程 = 一家公司（独立的"领地"）
线程 = 公司里的员工（共享公司资源，各自执行任务）
```

## 第3章：进程同步与互斥（最重要，常出大题！）

### 🔒 临界资源与临界区

**临界资源**：一次只允许一个进程使用的资源
（例：打印机、共享变量counter）

**临界区**：访问临界资源的那段代码

**进入临界区的4个原则（必背）：**
```
1. 空闲让进：临界区没人用时，要用的进程可以立即进入
2. 忙则等待：临界区有人用时，想进的进程要等
3. 有限等待：等待时间有上限，不能永远等
4. 让权等待：等不进去时，主动放弃CPU，不"忙等"
```

---

### 🚦 信号量机制（最重要！）

**信号量 = 整数变量 + P/V原语操作**

**P操作（申请资源，可能阻塞）：**
```c
P(S):
    S = S - 1
    if (S < 0):
        当前进程进入S的等待队列，挂起自己（阻塞）
```

**V操作（释放资源，可能唤醒）：**
```c
V(S):
    S = S + 1
    if (S <= 0):
        从S的等待队列中唤醒一个进程
```

**信号量S值的含义：**
```
S > 0：剩余可用资源数量
S = 0：资源用完，但没有进程在等
S < 0：|S| 个进程正在等待该资源
```

**互斥信号量（初值为1）—— 保证只有一个进程进临界区：**
```c
mutex = 1  // 互斥信号量，初值1表示临界资源可用

进程A：                进程B：
P(mutex)              P(mutex)
... 临界区 ...         ... 临界区 ...
V(mutex)              V(mutex)

若A在临界区，B执行P(mutex)后，mutex=-1<0，B被阻塞
A退出时执行V(mutex)，mutex=0≤0，唤醒等待的B
```

### 🏭 生产者-消费者问题（经典！）

**问题描述：**
```
有一个大小为n的缓冲区（仓库）
生产者：往缓冲区放产品
消费者：从缓冲区取产品
要求：
  - 缓冲区满时，生产者必须等
  - 缓冲区空时，消费者必须等
  - 同一时刻只有一个进程访问缓冲区（互斥）
```

**解法（3个信号量）：**
```c
semaphore mutex = 1;   // 互斥信号量，保护缓冲区
semaphore empty = n;   // 空缓冲区数量，初值=n（全空）
semaphore full = 0;    // 满缓冲区数量，初值=0（没有产品）

生产者进程：              消费者进程：
while(1) {               while(1) {
    生产一个产品;              P(full);      // 等待有产品
    P(empty);   // 等待空位     P(mutex);     // 进临界区
    P(mutex);   // 进临界区     从缓冲区取产品;
    放产品入缓冲区;             V(mutex);     // 出临界区
    V(mutex);   // 出临界区     V(empty);     // 空位+1
    V(full);    // 产品数+1     消费产品;
}                        }

⚠️ 重要：P操作的顺序！
  必须先做"同步P"（等资源），再做"互斥P"（进临界区）
  如果反过来（先互斥P，再同步P），可能造成死锁！
  
  错误写法：先P(mutex)再P(empty)
  若缓冲区满，生产者先占mutex，然后等empty；
  消费者要取产品，但mutex被占，等不进来，两者死锁！
```

### 📖 读者写者问题（经典！）

**问题：**
```
对一个共享文件：
  多个读者可以同时读（互不干扰）
  写者必须独占（写的时候不能有读者，也不能有其他写者）
```

**读者优先的解法：**
```c
int readcount = 0;    // 当前读者数量
semaphore mutex = 1;  // 保护readcount的互斥量
semaphore rw = 1;     // 读写互斥量

读者进程：              写者进程：
P(mutex);              P(rw);
readcount++;           ... 写 ...
if(readcount == 1)     V(rw);
    P(rw);   // 第一个读者锁住写者
V(mutex);

... 读 ...

P(mutex);
readcount--;
if(readcount == 0)
    V(rw);   // 最后一个读者解锁
V(mutex);

特点：读者优先——只要有读者在读，写者就一直等
缺点：可能导致写者"饥饿"（一直等不到）
```

### 💀 死锁

**死锁 = 多个进程互相等待对方持有的资源，永远无法继续**

**死锁的4个必要条件（全部同时成立才会死锁）：**
```
1. 互斥条件：资源只能被一个进程独占（如打印机）
2. 请求并持有：进程持有资源，同时还在申请新资源
3. 不可剥夺：进程持有的资源不能被强制取走，只能主动释放
4. 循环等待：存在环形等待链（A等B，B等C，C等A）

破坏任意一个条件，就不会死锁！
```

**银行家算法（死锁避免，常出大题！）：**

> 比喻：银行家放贷，每次放贷前要检查，保证剩余资金至少能让某些客户完成还款！

```
已知：
Max[i][j]：进程i最多需要j类资源多少个
Allocation[i][j]：进程i当前已分配到j类资源多少个
Available[j]：j类资源当前还剩多少可用
Need[i][j] = Max[i][j] - Allocation[i][j]（进程i还需要多少）

安全性检测算法（核心！）：
1. Work = Available（当前可用资源副本）
2. Finish[i] = false for all i（所有进程标记未完成）
3. 寻找满足条件的进程i：
     Finish[i] == false  且  Need[i] ≤ Work
   找到则：
     Work = Work + Allocation[i]（i完成，释放所有资源）
     Finish[i] = true
     回到步骤3
4. 若所有Finish[i] == true：当前状态安全！
   否则：不安全，可能死锁
```

**银行家算法例题：**
```
3种资源（A=10, B=5, C=7），5个进程P0~P4

           Max       Allocation    Need     Available
        A  B  C     A  B  C     A  B  C    A  B  C
P0     7  5  3     0  1  0     7  4  3    
P1     3  2  2     2  0  0     1  2  2    3  3  2
P2     9  0  2     3  0  2     6  0  0
P3     2  2  2     2  1  1     0  1  1
P4     4  3  3     0  0  2     4  3  1

检查是否安全：
初始Available = [3,3,2]
P1需要[1,2,2]≤[3,3,2]，可以运行！完成后释放[2,0,0]，Available=[5,3,2]
P3需要[0,1,1]≤[5,3,2]，可以运行！完成后释放[2,1,1]，Available=[7,4,3]
P4需要[4,3,1]≤[7,4,3]，可以运行！完成后释放[0,0,2]，Available=[7,4,5]
P0需要[7,4,3]≤[7,4,5]，可以运行！完成后释放[0,1,0]，Available=[7,5,5]
P2需要[6,0,0]≤[7,5,5]，可以运行！
安全序列：P1→P3→P4→P0→P2，状态安全！
```

## 第4章：处理机调度

### 📅 调度算法（重要！）

**① 先来先服务（FCFS）：**
```
规则：谁先来谁先用CPU，不抢占
优点：公平，简单
缺点：对短作业不利（"长作业"前面的短作业等很久）

例：进程  到达时间  服务时间
   P1     0        10
   P2     1        1
   P3     2        2
   
FCFS顺序：P1(0~10) → P2(10~11) → P3(11~13)
P2等了9秒，P3等了9秒，太可惜了（它们很快）
```

**② 短作业优先（SJF）：**
```
规则：选执行时间最短的作业（非抢占）
优点：平均等待时间最短（理论最优）
缺点：长作业可能永远等不到（饥饿）；需要预知执行时间

抢占式SJF（SRTF）：如果来了更短的作业，立即抢占当前进程
```

**③ 时间片轮转（RR）：**
```
规则：每个进程分配固定时间片（如10ms），时间片到就强制切换
优点：公平，响应快（适合交互式系统）
缺点：时间片大小影响性能
  时间片太大 → 退化为FCFS
  时间片太小 → 频繁切换，上下文切换开销大（浪费CPU）
```

**④ 优先级调度：**
```
规则：高优先级的进程先运行
静态优先级：创建时确定，不变
动态优先级：随时间调整（等待越久，优先级越高 → 避免饥饿）

老化机制：等待超过一定时间，优先级自动提升
```

**⑤ 多级反馈队列（最实用！）：**
```
多个优先级队列，时间片从高到低增大

队列1（最高优先级，时间片最短）：Q1
队列2：Q2（时间片是Q1的2倍）
队列3：Q3（时间片更大）
...

新进程 → 加入Q1
如果Q1时间片用完但未完成 → 降级到Q2
如果Q2时间片用完但未完成 → 降级到Q3
...

原理：短任务在Q1就能完成（高优先，短时间片）
      长任务虽然慢慢降级，但最终也能运行（公平）
```

## 第5章：内存管理

### 💾 内存分配方式

**连续分配（程序在内存中占连续的一块空间）：**
```
固定分区：把内存预先分成固定大小的区域
  缺点：产生"内部碎片"（区域没用完的部分）

动态分区：按需分配，大小不固定
  缺点：产生"外部碎片"（散落的小空闲区，放不下新程序）
  
  分配算法：
  首次适配（First Fit）：找第一个足够大的空闲区
  最佳适配（Best Fit）：找最小的足够大的空闲区（减少碎片，但产生很多小碎片）
  最坏适配（Worst Fit）：找最大的空闲区（减少小碎片的产生）
```

**分页（Paging）—— 离散分配：**
```
将内存和进程都划分为固定大小的"页"（通常4KB）
页表记录虚拟页号 → 物理帧号的映射

优点：无外部碎片（只有少量内部碎片，最多浪费一页）
缺点：页表占用内存，地址翻译开销

虚拟地址结构：[页号 | 页内偏移]
物理地址结构：[帧号 | 页内偏移]
```

**分段（Segmentation）—— 按逻辑划分：**
```
将程序按逻辑分成不等长的段：代码段、数据段、堆栈段...

虚拟地址：[段号 | 段内偏移]
段表：段号 → {段基址, 段长度}

优点：便于程序共享（共享某个段）、内存保护
缺点：产生外部碎片
```

## 第6章：文件系统

### 📁 文件的物理存储方式

**连续分配：**
```
文件占连续的磁盘块
优点：随机访问快（知道起始地址和大小，直接计算）
缺点：外部碎片；文件大小改变困难
类比：停车场划定连续车位
```

**链式分配：**
```
每个磁盘块有指向下一块的指针
优点：无外部碎片，文件可随意增长
缺点：只能顺序访问，不能随机访问
类比：链表
```

**索引分配：**
```
用一个索引块存储文件所有数据块的地址
直接索引：索引块直接指向数据块
多级索引：索引块→索引块→...→数据块（扩大文件容量）
优点：支持随机访问
缺点：对小文件浪费索引块
```

**UNIX的inode（考点！）：**
```
inode = 文件的元数据（大小、权限、时间等）+ 数据块地址

inode中包含：
  10个直接块地址（直接指向数据块）
  1个一级间接块（块中存地址，再指向数据块）
  1个二级间接块（两层间接）
  1个三级间接块（三层间接）

设每块4KB，每个地址4字节，则每块可存1024个地址：
  直接块：10 × 4KB = 40KB
  一级间接：1024 × 4KB = 4MB
  二级间接：1024² × 4KB = 4GB
  三级间接：1024³ × 4KB = 4TB
  理论最大文件：很大！
```

## 第7章：磁盘调度

### 💿 磁盘访问时间的计算

```
磁盘访问时间 = 寻道时间 + 旋转延迟 + 数据传输时间

寻道时间：磁头移动到目标磁道的时间（最长！是优化重点）
旋转延迟：等待目标扇区转到磁头下的时间（平均 = 半圈时间）
传输时间：数据读写的时间（通常较短）
```

**磁盘调度算法：**
```
FCFS：先来先服务（公平但可能效率低，磁头来回乱跳）

SSTF（最短寻道时间优先）：选离磁头最近的请求
  优点：平均寻道时间短
  缺点：外围磁道可能饥饿

SCAN（电梯算法）★★★：
  磁头沿一个方向移动，扫描所有请求，到达最远端再折返
  类比：电梯——上行时停所有上行请求，下行时停所有下行请求
  优点：公平，效率好，无饥饿

C-SCAN（循环扫描）：
  只向一个方向移动，到达最远端后快速返回起点，不处理返回途中的请求
  优点：对两端的请求更公平（SCAN会让某端等更久）

LOOK/C-LOOK：
  与SCAN/C-SCAN类似，但到最远的请求处就折返（不必到磁盘物理边界）
  更实际！不会做无用功
```

**SCAN算法例题：**
```
磁头当前在53号磁道，向大方向移动
请求队列：98, 183, 37, 122, 14, 124, 65, 67

排序后分两段：
  向大方向：65, 67, 98, 122, 124, 183
  向小方向：37, 14

SCAN访问顺序：53→65→67→98→122→124→183→37→14

总移动距离 = (183-53) + (183-14) = 130 + 169 = 299（比FCFS少很多）
```

## 第1章：计算机网络概述

### 🌐 基本概念与性能指标

**时延的计算（重要！）：**
```
端到端总时延 = 发送时延 + 传播时延 + 处理时延 + 排队时延

发送时延（传输时延）：把数据"发"出去需要的时间
  = 数据帧长度（bit） / 信道带宽（bps）
  例：100MB的文件，1Gbps的链路
  发送时延 = 100×8×10^6 bit / 10^9 bps = 0.8秒

传播时延：信号从发送方传到接收方的时间
  = 信道长度 / 信号传播速度（约2×10^8 m/s，光速的2/3）
  例：1000km的光纤
  传播时延 = 10^6 m / (2×10^8 m/s) = 5ms

带宽时延积 = 传播时延 × 带宽
  = 信道中"在途"的数据量（"管道"的容量）

RTT（往返时延）= 2 × 传播时延
  （从发出到收到确认的时间）
```

## 第3章：数据链路层

### 🔗 滑动窗口协议（重要！）

**为什么需要滑动窗口？**

```
简单的停止等待：发一帧，等确认，再发下一帧
  效率极低！信道大部分时间空闲

滑动窗口：发送方可以连续发送多帧，不必每帧都等确认
  信道利用率大大提高！
```

**三种协议对比：**
```
停止等待协议（Stop-and-Wait, SW）：
  窗口大小 = 1（发一个等一个）
  信道利用率 = 1 / (1 + 2a)，其中 a = 传播时延/发送时延
  
后退N帧协议（Go-Back-N, GBN）：
  发送窗口 = 2^n - 1（n是序号字段位数）
  接收窗口 = 1
  出错时：重传出错帧及其后所有帧（后退N帧）
  信道利用率 = W/(1+2a)（W=窗口大小，若W≥1+2a则利用率=1）
  
选择重传协议（Selective Repeat, SR）：
  发送窗口 = 接收窗口 = 2^(n-1)
  出错时：只重传出错的帧（效率更高，但需要缓冲区）

窗口大小口诀：
  GBN：发送窗口 ≤ 2^n - 1
  SR：发送窗口 ≤ 2^(n-1)
  （原因：防止序号混乱，接收方无法区分旧帧和新帧）
```

---

### 🖥️ 以太网（CSMA/CD）

**CSMA/CD工作原理：**
```
"先听后发，边发边听，碰撞停发，随机重发"

步骤详解：
1. 先听（Carrier Sense）：发送前监听信道，空闲才发
2. 边发边听（Collision Detection）：发送时同时检测
3. 碰撞：若检测到碰撞，立即停发，发阻塞信号
4. 随机退避：等待随机时间后重发（指数退避算法）
5. 最多重发16次，仍失败则报错

为什么要有最小帧长？
  帧必须足够长，使得发送完毕前能检测到碰撞
  最小帧长 = 2 × 最大传播时延 × 带宽
  以太网最小帧：64字节（512bit）
```

**以太网帧格式：**
```
[前导符(7B)] [帧开始定界(1B)] [目的MAC(6B)] [源MAC(6B)] [类型(2B)] [数据(46-1500B)] [FCS(4B)]

前导符：10101010...用于同步时钟
FCS：帧校验序列（CRC），用于检测错误
最小帧数据：46B，最大：1500B（MTU）
```

**MAC地址：**
```
48位（6字节），如 AA:BB:CC:DD:EE:FF
前3字节：厂商OUI（由IEEE分配）
后3字节：设备编号（厂商自己分配）

特殊地址：
  FF:FF:FF:FF:FF:FF → 广播地址
  最低位=0 → 单播；最低位=1 → 多播
```

## 第4章：网络层

### 🌐 IP地址与子网划分（必考！）

**IP地址分类（传统分类）：**
```
A类：第一字节0-127（最高位是0）
     网络号7位，主机号24位
     可用主机数：2^24 - 2 = 约1600万
     （-2是因为全0是网络地址，全1是广播地址）

B类：第一字节128-191（最高两位是10）
     网络号14位，主机号16位
     可用主机数：2^16 - 2 = 65534

C类：第一字节192-223（最高三位是110）
     网络号21位，主机号8位
     可用主机数：2^8 - 2 = 254

D类：224-239（多播地址，用于一对多通信）
E类：240-255（保留）

记忆：A≤127，B128~191，C192~223，D多播，E保留
```

**特殊IP地址（必背）：**
```
127.0.0.1：本地回环地址，测试本机网络栈
0.0.0.0：代表本机所有IP或"任意IP"
255.255.255.255：受限广播（发到本网络所有主机）
主机号全0（如192.168.1.0）：网络地址（不分配给主机）
主机号全1（如192.168.1.255）：定向广播地址

私有地址（局域网内使用，Internet上不路由）：
10.0.0.0/8         （10.x.x.x）
172.16.0.0/12       （172.16.x.x ~ 172.31.x.x）
192.168.0.0/16      （192.168.x.x）
```

**CIDR子网划分（必考计算！）：**
```
格式：IP地址/前缀长度（如 192.168.1.0/24）

子网掩码：前缀长度个1，其余补0
/24 → 11111111.11111111.11111111.00000000 = 255.255.255.0

计算：
网络地址 = IP地址 AND 子网掩码
广播地址 = 网络地址 OR (NOT 子网掩码)
可用主机数 = 2^(32-前缀长度) - 2

例：192.168.1.130/25
  子网掩码：/25 = 255.255.255.128（11111111 11111111 11111111 10000000）
  130的二进制：10000010
  网络地址：192.168.1.128（130 AND 128 = 128，因为10000010 AND 10000000 = 10000000）
  广播地址：192.168.1.255（128 OR 127 = 255，因为10000000 OR 01111111 = 11111111）
  可用主机：2^7 - 2 = 126台
```

**子网划分例题：**
```
将192.168.1.0/24（C类）划分为4个子网：

借2位（2²=4）→ 前缀从/24变为/26
子网掩码：255.255.255.192（/26）

4个子网（后8位从00,01,10,11开始）：
  192.168.1.0/26    网络地址，广播地址192.168.1.63，主机1~62
  192.168.1.64/26   网络地址，广播地址192.168.1.127，主机65~126
  192.168.1.128/26  网络地址，广播地址192.168.1.191，主机129~190
  192.168.1.192/26  网络地址，广播地址192.168.1.255，主机193~254
```

### 🤝 TCP三次握手（必背！）

```
为什么需要三次？建立可靠连接，确认双方都能收发！

客户端                              服务端
   |                                  |
   |──SYN(seq=x)──────────────────→  |  第1次：客户端说"我要连你"
   |                                  |
   |←─SYN+ACK(seq=y, ack=x+1)──────  |  第2次：服务端说"好的，我也准备好了"
   |                                  |
   |──ACK(ack=y+1)─────────────────→  |  第3次：客户端确认"好，我知道了"
   |                                  |
   |======== 连接建立，开始传输 ========|

为什么不是两次？
  若客户端发的SYN在网络中延迟，客户端超时重发，
  最终建立连接后完成，但延迟的SYN到达服务端，
  服务端以为是新连接请求，建立新连接，等待客户端数据，
  但客户端不理它，服务端白白浪费资源。
  三次握手：服务端收到客户端的ACK才建立连接，避免此问题！
```

---

### 👋 TCP四次挥手（必背！）

```
为什么四次？TCP是全双工，每个方向独立关闭！

主动关闭方(A)                       被动关闭方(B)
   |                                   |
   |──FIN(seq=u)──────────────────→    |  A说"我不再发数据了"
   |                                   |
   |←─ACK(ack=u+1)────────────────     |  B确认（B还可以发数据！）
   |                                   |  B继续发剩余数据...
   |                                   |
   |←─FIN(seq=v)──────────────────     |  B说"我也不发了"
   |                                   |
   |──ACK(ack=v+1)─────────────────→   |  A确认
   |                                   |
   A等待2MSL后关闭                       B收到ACK立即关闭

为什么等待2MSL（最大报文段寿命）？
  如果最后一个ACK丢失，B会重传FIN，
  A需要能收到重传的FIN并重发ACK，
  2MSL保证A的ACK能到达B，且B的重传FIN也能到达A
```

### 📊 TCP流量控制与拥塞控制

**流量控制（接收方控制发送方的速率）：**
```
接收方在ACK中告知发送方自己的接收窗口大小（rwnd）
发送方发送的数据量不超过rwnd

若接收缓冲区满（rwnd=0）：
  发送方停止发送，周期性发送零窗口探测报文，等接收方更新窗口
```

**拥塞控制（全局控制，防止网络过载）：**
```
发送方维护拥塞窗口（cwnd），实际发送量 = min(cwnd, rwnd)

四个算法：

1. 慢开始（Slow Start）：
   cwnd从1开始，每个RTT翻倍（1,2,4,8...）
   虽然叫"慢"开始，实际是指数增长，一点不慢！
   直到cwnd达到慢开始阈值（ssthresh）

2. 拥塞避免（Congestion Avoidance）：
   cwnd每个RTT只+1（线性增长，谨慎增加）
   直到检测到拥塞（超时或收到3个重复ACK）

3. 快重传（Fast Retransmit）：
   收到3个连续的重复ACK，立即重传（不等超时）

4. 快恢复（Fast Recovery）：
   收到3个重复ACK时（说明只是轻微拥塞，不是超时）：
   ssthresh = cwnd / 2
   cwnd = ssthresh（而不是从1开始！）
   进入拥塞避免阶段

超时（严重拥塞）时：
   ssthresh = cwnd / 2
   cwnd = 1
   重新慢开始
```

**拥塞控制图示：**
```
cwnd（拥塞窗口）
    │
16  │              *
    │             /|
 8  │            / |←拥塞避免（线性增长）
    │           /  |
 4  │          /   |
    │    慢开始│   拥塞！超时
 2  │   (指数) │   ssthresh=8→cwnd=1
    │         │   重新慢开始
 1  │  *      │
    └──────────────────→ 时间（RTT数）
    
（若是快重传/快恢复：cwnd不从1开始，而是从ssthresh开始拥塞避免）
```

### 🌍 DNS域名解析过程

```
目标：将域名 www.example.com 解析为IP地址

步骤：
1. 查本机DNS缓存 → 找到直接返回
2. 查hosts文件（如/etc/hosts）→ 找到直接返回
3. 向本地DNS服务器（ISP提供）查询
4. 本地DNS没有：
   ① 迭代查询：
      → 问根DNS服务器：".com"由谁负责？
      → 根DNS返回：.com的顶级域名服务器地址
      → 问.com的顶级域名服务器：example.com由谁负责？
      → 返回：example.com的权威DNS地址
      → 问example.com的权威DNS：www.example.com的IP是多少？
      → 返回IP地址
   ② 递归查询：让DNS服务器替你问到底
5. 本地DNS把结果缓存，返回给用户

DNS记录类型：
A记录：域名 → IPv4地址（最常用）
AAAA记录：域名 → IPv6地址
CNAME：别名 → 真实域名（如www→example.com）
MX：邮件服务器
NS：该域的域名服务器
```

### 🌐 HTTP协议

```
请求方法：
GET：获取资源（参数在URL中，可见，有长度限制）
POST：提交数据（参数在消息体中，不可见，无长度限制）
PUT：创建或更新资源
DELETE：删除资源
HEAD：只要头部（不要响应体）

状态码（必背！）：
1xx：信息性（100 Continue）
2xx：成功（200 OK, 201 Created）
3xx：重定向（301永久重定向, 302临时重定向, 304未修改）
4xx：客户端错误（400坏请求, 401未授权, 403禁止, 404找不到）
5xx：服务器错误（500内部错误, 503服务不可用）

HTTP版本演进：
HTTP/1.0：每次请求都要建立TCP连接（短连接）
HTTP/1.1：持久连接（默认），管道化（可以发多个请求再等响应）
HTTP/2.0：二进制协议，多路复用，头部压缩，服务器推送
HTTP/3.0：基于UDP的QUIC协议，解决了TCP的队头阻塞问题
```
