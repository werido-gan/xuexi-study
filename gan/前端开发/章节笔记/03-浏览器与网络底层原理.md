## 3. 浏览器与网络底层原理（进阶）

### 3.1 浏览器渲染流程（Critical Rendering Path）

**类比**：浏览器渲染就像建房子。先搭骨架（DOM），再制定设计图（CSSOM），再合并成完整方案（Render Tree），再测量精确位置（Layout），最后刷漆完工（Paint）。

```
[关键渲染路径（Critical Rendering Path）]

HTML字节流
    │
    ▼
词法分析（Tokenization）→ Token流
    │
    ▼
构建 DOM 树（Document Object Model）
    │
    │                 CSS字节流
    │                     │
    │                     ▼
    │              构建 CSSOM 树
    │                     │
    └──────────┬──────────┘
               │
               ▼
         合并 Render Tree
        （只包含可见节点，
          display:none 的不包含）
               │
               ▼
           Layout（布局/回流）
         计算每个元素的精确位置和大小
               │
               ▼
         Paint（绘制/重绘）
         将像素绘制到屏幕上
               │
               ▼
         Composite（合成）
         GPU将各层合并输出到屏幕
```

**重要概念：渲染阻塞**

```
HTML 解析遇到 <script>（无defer/async）→ 暂停解析 → 下载并执行JS → 继续解析
CSS 文件 → 阻塞渲染（CSSOM构建完才能构建Render Tree）

[script 加载策略]

<script src="a.js"></script>        普通：阻塞HTML解析
                                     ─解析─|下载+执行|─继续解析─

<script defer src="a.js"></script>  defer：不阻塞解析，HTML解析完再执行
                                     ─解析─────────────────|执行|─DOMContentLoaded

<script async src="a.js"></script>  async：下载不阻塞，下载完立即执行
                                     ─解析─|执行|─────────── (执行时机不确定)
```

> **最佳实践**：
> - `<script>` 放在 `</body>` 前（防止阻塞）
> - 或使用 `defer` 属性
> - CSS `<link>` 放在 `<head>` 中（尽早加载）

### 3.2 回流与重绘

**类比**：
- **回流（Reflow）** = 重新装修，要重新测量、移家具。代价极高。
- **重绘（Repaint）** = 重新刷漆，位置没变，只是颜色变了。代价较低。

```
[触发回流的操作（代价大）]
- 改变元素尺寸（width/height/padding/margin）
- 改变元素位置（top/left）
- 添加/删除DOM元素
- 改变字体大小
- 窗口resize
- 读取布局属性（offsetTop/scrollTop/clientWidth...）
  → 浏览器为了返回准确值，会强制触发回流！

[触发重绘的操作（代价小）]
- 改变颜色（color/background-color）
- 改变visibility
- 改变outline
```

**性能优化技巧：**

```javascript
// ❌ 错误：多次读写布局属性，每次都触发回流
for (let i = 0; i < 1000; i++) {
  element.style.left = element.offsetLeft + 1 + 'px';
  // offsetLeft 强制回流 → style.left 触发回流 → 循环1000次 = 2000次回流
}

// ✅ 正确：批量读取，批量写入
const startLeft = element.offsetLeft;  // 读取一次
for (let i = 0; i < 1000; i++) {
  element.style.left = startLeft + i + 1 + 'px';
}

// ✅ 更好：使用 CSS transform，不触发回流
element.style.transform = `translateX(${startLeft + 1000}px)`;

// ✅ 使用 DocumentFragment 批量操作DOM
const fragment = document.createDocumentFragment();
for (let i = 0; i < 100; i++) {
  const li = document.createElement('li');
  li.textContent = `Item ${i}`;
  fragment.appendChild(li);  // 操作内存中的fragment，不触发回流
}
list.appendChild(fragment);  // 只触发一次回流
```

### 3.3 事件循环（Event Loop）⭐ 面试必考

**为什么 JavaScript 是单线程的？**

JS 最初设计用于浏览器，如果多线程同时操作 DOM，会产生竞争条件（Race Condition）。所以 Brendan Eich 设计时选择了单线程。但单线程如何处理异步？这就是事件循环（Event Loop）。

```
[事件循环完整图解]

    ┌─────────────────────────────────────┐
    │           调用栈（Call Stack）        │
    │   同步代码在这里执行                  │
    │                                     │
    │  ┌───────┐                          │
    │  │console│                          │
    │  ├───────┤                          │
    │  │ fn2() │                          │
    │  ├───────┤                          │
    │  │ fn1() │                          │
    │  ├───────┤                          │
    │  │main() │                          │
    │  └───────┘                          │
    └────────────────┬────────────────────┘
                     │ 调用栈为空时
                     │ Event Loop检查
                     ▼
    ┌────────────────────────────────────┐
    │          消息队列（Task Queue）      │
    │                                    │
    │  微任务队列（Microtask Queue）       │
    │  优先级更高！每次宏任务后先清空      │
    │  ┌─────────────────────────────┐   │
    │  │ Promise.then / queueMicrotask│   │
    │  │ MutationObserver             │   │
    │  └─────────────────────────────┘   │
    │                                    │
    │  宏任务队列（Macrotask Queue）       │
    │  ┌─────────────────────────────┐   │
    │  │ setTimeout / setInterval    │   │
    │  │ I/O 事件                    │   │
    │  │ requestAnimationFrame       │   │
    │  └─────────────────────────────┘   │
    └────────────────────────────────────┘
                     │
    ┌────────────────────────────────────┐
    │           Web APIs                 │
    │  浏览器提供的异步能力               │
    │  (setTimeout/fetch/DOM事件...)     │
    └────────────────────────────────────┘
```

**事件循环执行顺序：**

```javascript
console.log('1');  // 同步

setTimeout(() => {
  console.log('2');  // 宏任务
}, 0);

Promise.resolve().then(() => {
  console.log('3');  // 微任务
});

console.log('4');  // 同步

// 输出顺序: 1 → 4 → 3 → 2
// 原因: 
// 1. 同步: 1, 4
// 2. 微任务（清空）: 3
// 3. 宏任务: 2
```

**经典面试题解析：**

```javascript
async function async1() {
  console.log('async1 start');    // 2
  await async2();
  console.log('async1 end');      // 6（微任务）
}

async function async2() {
  console.log('async2');          // 3
}

console.log('script start');     // 1

setTimeout(() => {
  console.log('setTimeout');     // 8（宏任务）
}, 0);

async1();

new Promise((resolve) => {
  console.log('promise1');        // 4（Promise构造函数同步执行）
  resolve();
}).then(() => {
  console.log('promise2');        // 7（微任务）
});

console.log('script end');       // 5

// 输出: script start → async1 start → async2 → promise1 
//       → script end → async1 end → promise2 → setTimeout
```

> **理解关键**：`await` 后面的代码相当于放入微任务队列。执行到 `await async2()` 时，先执行 `async2()`，然后 `async1 end` 被放入微任务队列，继续执行后续同步代码。

### 3.4 TCP/IP 原理简述

```
[网络分层模型]

应用层   │ HTTP / HTTPS / FTP / DNS
         │ 负责：应用间通信协议
─────────┼────────────────────────
传输层   │ TCP / UDP
         │ 负责：端到端可靠传输
         │ TCP：三次握手，保证可靠
         │ UDP：不保证可靠，但速度快
─────────┼────────────────────────
网络层   │ IP / ICMP
         │ 负责：路由寻址（IP地址）
─────────┼────────────────────────
数据链路 │ Ethernet / WiFi
层       │ 负责：相邻节点间传输
─────────┼────────────────────────
物理层   │ 电缆 / 光纤 / 无线电波
         │ 负责：0和1的物理传输
```

**TCP 三次握手：**

```
客户端                    服务器
  │── SYN（我想连接你）──►  │
  │◄── SYN+ACK（好的，我也想连接你）──  │
  │── ACK（好的，连接建立）──► │
  │                         │
  │  开始传输数据              │
```

### 3.5 第三章总结

```
[浏览器原理核心知识]

浏览器底层
├── 渲染流程
│   └── HTML→DOM→CSSOM→Render Tree→Layout→Paint→Composite
├── 回流 vs 重绘
│   ├── 回流：几何属性改变，代价高
│   └── 重绘：视觉属性改变，代价低
├── 事件循环
│   ├── 调用栈（同步）
│   ├── 微任务（Promise.then，优先）
│   └── 宏任务（setTimeout）
└── TCP/IP
    └── 五层模型，三次握手建连接
```
