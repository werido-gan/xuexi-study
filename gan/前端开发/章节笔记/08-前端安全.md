## 8. 前端安全

### 8.1 XSS（跨站脚本攻击）

**类比**：XSS 就像有人在你的留言本上偷偷夹了一张会"自动执行任务"的纸条，任何看到这张纸条的人都会被影响。

```
[XSS 攻击类型]

存储型XSS（最危险）:
  攻击者 → 提交含恶意脚本的评论
  → 存入数据库
  → 其他用户访问页面
  → 浏览器执行脚本（窃取Cookie、冒充用户操作）

反射型XSS:
  攻击者 → 构造含脚本的URL
  → 诱导受害者点击
  → 服务器将URL参数反射到页面
  → 浏览器执行脚本

DOM型XSS:
  前端代码: document.innerHTML = location.hash
  → 攻击者构造 /#<img src=x onerror=alert(document.cookie)>
  → 浏览器执行脚本
```

**防御措施：**

```javascript
// 1. 对输出进行HTML转义（最关键！）
function escapeHtml(str) {
  return str
    .replace(/&/g, '&amp;')
    .replace(/</g, '&lt;')
    .replace(/>/g, '&gt;')
    .replace(/"/g, '&quot;')
    .replace(/'/g, '&#039;');
}

// 2. 使用安全的API（不用innerHTML）
// ❌ 危险
element.innerHTML = userInput;

// ✅ 安全
element.textContent = userInput;  // 自动转义

// 3. CSP（Content Security Policy）内容安全策略
// 在HTTP响应头设置
Content-Security-Policy: 
  default-src 'self';
  script-src 'self' 'nonce-xxx';  // 只允许有nonce的脚本
  style-src 'self' 'unsafe-inline';

// 4. Cookie 设置 HttpOnly（JS无法读取Cookie）
Set-Cookie: session=xxx; HttpOnly; Secure; SameSite=Strict
```

### 8.2 CSRF（跨站请求伪造）

**类比**：你已经登录了银行网站，攻击者诱导你访问他的恶意网站，该网站悄悄向银行发送了转账请求（利用你的登录状态）。

```
[CSRF 攻击流程]

1. 用户登录 bank.com（获得Cookie）
2. 用户访问 evil.com
3. evil.com 页面有: <img src="bank.com/transfer?to=hacker&amount=10000">
4. 浏览器自动带上 bank.com 的 Cookie 发送请求
5. 银行服务器认为是合法请求，转账成功！
```

**防御措施：**

```javascript
// 1. CSRF Token（主流方案）
// 服务器生成随机token，存在Session中，返回给前端
// 前端每次请求携带这个token
// 服务器验证token是否匹配

// Axios 拦截器自动加CSRF Token
axios.interceptors.request.use(config => {
  const token = getCsrfToken();  // 从Cookie或meta标签读取
  config.headers['X-CSRF-Token'] = token;
  return config;
});

// 2. SameSite Cookie（现代浏览器首选）
Set-Cookie: session=xxx; SameSite=Strict
// Strict: 第三方网站请求完全不带Cookie（最安全，但可能影响体验）
// Lax: 大多数情况不带，GET导航除外
// None: 都带（需同时设置Secure，即HTTPS）

// 3. 验证 Referer / Origin
// 服务器检查请求来源是否是合法域名
if (request.headers.origin !== 'https://bank.com') {
  return 403;
}

// 4. 敏感操作要求二次验证（密码/短信）
```
