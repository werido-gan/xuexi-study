## 7. 性能优化（详细）

### 7.1 Web Vitals 核心指标

Google 定义的用户体验关键指标：

```
[Core Web Vitals]

LCP（Largest Contentful Paint，最大内容绘制）
  衡量：页面主要内容加载速度
  目标：≤ 2.5秒（良好）
  优化：优先加载关键资源，图片懒加载

FID（First Input Delay，首次输入延迟）
  衡量：用户第一次交互到浏览器响应的时间
  目标：≤ 100ms
  优化：减少主线程阻塞，代码分割

CLS（Cumulative Layout Shift，累计布局偏移）
  衡量：页面视觉稳定性（元素跳动）
  目标：≤ 0.1
  优化：图片/广告预留空间，避免动态内容插入

INP（Interaction to Next Paint，新增）
  衡量：整体响应性（替代FID）
  目标：≤ 200ms
```

### 7.2 首屏优化

```
[首屏时间线]

导航开始 → DNS → TCP → TLS → 请求 → 响应 → HTML解析 → CSS/JS → 渲染 → 可交互

优化每个环节：

1. 减少 DNS 查询: dns-prefetch
   <link rel="dns-prefetch" href="//cdn.example.com">

2. 提前建立连接: preconnect
   <link rel="preconnect" href="//api.example.com">

3. HTTP/2 + CDN: 并行请求 + 就近分发

4. 关键 CSS 内联: 避免首屏渲染等待CSS文件加载
   <style>/* 首屏关键CSS直接内联 */</style>

5. SSR / SSG: 服务端渲染，直接返回完整HTML
```

### 7.3 懒加载 / 预加载

```javascript
// === 图片懒加载 ===

// 方法1: HTML原生懒加载（推荐，简单）
<img src="photo.jpg" loading="lazy" alt="photo">

// 方法2: Intersection Observer API（可控制更多逻辑）
const observer = new IntersectionObserver((entries) => {
  entries.forEach(entry => {
    if (entry.isIntersecting) {
      const img = entry.target;
      img.src = img.dataset.src;   // 真实地址存在 data-src
      observer.unobserve(img);     // 加载后停止观察
    }
  });
}, { threshold: 0.1 });  // 10%可见时触发

document.querySelectorAll('[data-src]').forEach(img => {
  observer.observe(img);
});

// === 路由懒加载（React）===
import { lazy, Suspense } from 'react';

// 不要全部导入，用到时再加载
const HomePage = lazy(() => import('./pages/HomePage'));
const ProductPage = lazy(() => import('./pages/ProductPage'));

function App() {
  return (
    <Suspense fallback={<Loading />}>  {/* 加载中显示Loading */}
      <Routes>
        <Route path="/" element={<HomePage />} />
        <Route path="/products" element={<ProductPage />} />
      </Routes>
    </Suspense>
  );
}

// === 资源预加载 ===

// preload: 当前页面需要，提前加载
<link rel="preload" href="critical-font.woff2" as="font" crossorigin>

// prefetch: 下个页面可能需要，利用空闲时间提前获取
<link rel="prefetch" href="/next-page.js">
```

### 7.4 代码分割

```javascript
// === Webpack 代码分割策略 ===

// 1. 入口分割（多页面应用）
module.exports = {
  entry: {
    main: './src/index.js',
    vendor: './src/vendor.js'  // 第三方库单独打包
  }
};

// 2. 动态导入（最灵活）
button.addEventListener('click', async () => {
  const { default: Chart } = await import('./Chart.js');  // 点击才加载
  new Chart(data);
});

// 3. SplitChunks（自动提取公共模块）
optimization: {
  splitChunks: {
    chunks: 'all',
    cacheGroups: {
      vendor: {
        test: /[\\/]node_modules[\\/]/,
        name: 'vendor',     // react、lodash等打包到vendor.js
        priority: 10
      },
      common: {
        minChunks: 2,       // 被2个以上chunk引用的模块提取
        name: 'common',
        priority: 5
      }
    }
  }
}
```

### 7.5 缓存策略

```
[浏览器缓存层次]

Service Worker缓存（最高优先级，可编程控制）
    │
    ▼
内存缓存（Memory Cache，关闭标签页失效）
    │
    ▼
磁盘缓存（Disk Cache，根据HTTP头决定）
    │
    ▼
网络请求

[HTTP 缓存策略]

强缓存（不发请求直接用缓存）:
Cache-Control: max-age=31536000  → 缓存1年（适合带hash的静态资源）
Cache-Control: no-cache          → 每次都要验证（不是不缓存！）
Cache-Control: no-store          → 不缓存（敏感数据）

协商缓存（发请求验证是否可用缓存）:
ETag: "abc123"                   → 内容hash
Last-Modified: Mon, 1 Jan 2024   → 最后修改时间
→ 服务器返回304 Not Modified，浏览器使用本地缓存
```

```
[缓存策略最佳实践]

静态资源（JS/CSS/图片）:
  文件名包含contenthash（如 main.abc123.js）
  Cache-Control: max-age=31536000 (1年)
  内容改变 → 文件名变 → 缓存失效

HTML文件:
  Cache-Control: no-cache（每次验证，确保拿到最新入口）

API响应:
  敏感数据: Cache-Control: no-store
  公共数据: Cache-Control: max-age=300 (5分钟)
```

### 7.6 性能优化清单

```
[前端性能优化全清单]

加载优化
├── CDN加速（就近分发）
├── HTTP/2（多路复用）
├── Gzip/Brotli压缩
├── 图片优化（WebP格式，适当压缩）
├── 懒加载（图片、路由、组件）
└── 预加载（preload, prefetch）

渲染优化
├── 避免不必要的回流重绘
├── 使用transform/opacity动画（触发GPU合成）
├── 虚拟列表（大数据列表）
└── 防抖节流（高频事件）

代码优化
├── Tree Shaking（删除未使用代码）
├── 代码分割（按需加载）
├── 减少包体积（分析bundle，替换重型库）
└── 使用Web Worker处理CPU密集任务

缓存优化
├── 浏览器强缓存（contenthash）
├── Service Worker（离线能力）
└── 接口缓存（SWR/React Query）
```
