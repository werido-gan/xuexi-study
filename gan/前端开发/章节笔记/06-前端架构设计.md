## 6. 前端架构设计（高级）

### 6.1 组件化设计

**组件化思想**：把页面拆分成一个个可复用的积木块（组件），每块积木只负责自己的事情。

```
[组件化层次]

页面（Page）
  ├── 业务组件（Business Components）
  │     ├── 用户卡片（UserCard）
  │     └── 商品列表（ProductList）
  │           └── 商品项（ProductItem）
  │
  └── 基础组件（Base Components / Design System）
        ├── Button
        ├── Input
        ├── Modal
        └── Table
```

**组件设计原则：**

```
[好的组件设计原则]

1. 单一职责（SRP）
   每个组件只做一件事
   ❌ UserCard（既显示用户信息，又处理购物车，又发请求）
   ✅ UserCard（只负责显示用户信息）

2. 开放封闭（OCP）
   对扩展开放，对修改封闭
   通过 Props/Slots 让组件可配置，而不是修改源码

3. 受控 vs 非受控
   受控组件：数据由父组件控制（通过props传入，onChange回调）
   非受控组件：数据由自身管理（useRef读取值）

4. 容器组件 vs 展示组件（智能/傻瓜）
   容器组件：处理数据、状态、API请求
   展示组件：只管UI渲染，数据通过props传入
```

**组件通信模式：**

```
[组件通信方式]

父 → 子: Props（向下传递数据）
子 → 父: 回调函数/事件（向上传递事件）
兄弟间: 状态提升（lift state up）到共同父组件
跨层级: Context / 状态管理库（Redux/Pinia）
```

### 6.2 状态管理

#### 为什么需要状态管理？

```
[没有状态管理的痛苦]

                UserPage
                    │
              ┌─────┴─────┐
           Header       CartIcon
              │               │
           NavMenu         CartCount  ← 这两个组件都需要"用户信息"
```

**当组件树很深，Props 传递变成噩梦（Props Drilling）**

#### Redux（React 生态）

```
[Redux 数据流（单向数据流）]

用户操作（Click）
    │
    ▼
dispatch(action)
    │  action = { type: 'ADD_TO_CART', payload: item }
    ▼
Reducer（纯函数，根据action更新state）
    │  newState = reducer(oldState, action)
    ▼
Store（存储全局状态）
    │
    ▼
Subscribe（视图订阅Store变化）
    │
    ▼
React 重渲染（UI更新）
```

```javascript
// Redux Toolkit（现代Redux写法，推荐）
import { createSlice, configureStore } from '@reduxjs/toolkit';

// Slice（状态片段）
const cartSlice = createSlice({
  name: 'cart',
  initialState: {
    items: [],
    total: 0
  },
  reducers: {
    addItem(state, action) {
      state.items.push(action.payload);    // RTK内置Immer，可直接"修改"
      state.total += action.payload.price;
    },
    removeItem(state, action) {
      state.items = state.items.filter(item => item.id !== action.payload);
    }
  }
});

export const { addItem, removeItem } = cartSlice.actions;

// Store
const store = configureStore({
  reducer: {
    cart: cartSlice.reducer,
    user: userSlice.reducer
  }
});

// React组件中使用
function CartButton({ item }) {
  const dispatch = useDispatch();
  const cartCount = useSelector(state => state.cart.items.length);
  
  return (
    <button onClick={() => dispatch(addItem(item))}>
      加入购物车 ({cartCount})
    </button>
  );
}
```

#### Zustand（轻量级，推荐中小项目）

```javascript
// Zustand：极其简洁
import { create } from 'zustand';

const useCartStore = create((set, get) => ({
  items: [],
  total: 0,
  
  addItem: (item) => set(state => ({
    items: [...state.items, item],
    total: state.total + item.price
  })),
  
  removeItem: (id) => set(state => ({
    items: state.items.filter(i => i.id !== id)
  })),
  
  clearCart: () => set({ items: [], total: 0 })
}));

// 使用（不需要Provider！）
function CartIcon() {
  const { items, addItem } = useCartStore();
  return <span>购物车 ({items.length})</span>;
}
```

### 6.3 微前端架构

**什么是微前端？**

**类比**：大型电商网站（淘宝、京东）的首页、商品详情页、购物车、个人中心，这几个功能可能是不同团队开发的，用不同的技术栈，但最终合并在一个网站里。

```
[微前端架构图]

┌──────────────────────────────────────────────────────┐
│                    主应用（Shell App）                 │
│         负责：路由、鉴权、公共组件、全局状态             │
│                                                      │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐  │
│  │  子应用A     │  │  子应用B     │  │  子应用C     │  │
│  │  （React）   │  │  （Vue）     │  │  （Angular） │  │
│  │  商品模块    │  │  购物车模块   │  │  后台管理    │  │
│  └─────────────┘  └─────────────┘  └─────────────┘  │
│                                                      │
└──────────────────────────────────────────────────────┘

特点：
- 各子应用独立开发、独立部署
- 技术栈不受限制
- 团队自治，互不影响
```

**实现方案：**

```
[微前端实现方案对比]

iframe（最简单，隔离最彻底）
  优点：完全隔离，简单
  缺点：通信困难，性能差，UI割裂

qiankun（蚂蚁金服出品，基于single-spa）
  优点：成熟，支持多框架
  缺点：样式隔离靠沙箱，有一定复杂度

Module Federation（Webpack 5内置）
  优点：构建时集成，性能好
  缺点：需要Webpack 5，版本管理复杂

Wujie（腾讯出品，基于iframe改进）
  优点：CSS隔离彻底，JS沙箱
  缺点：较新，生态尚在发展
```

### 6.4 Monorepo

**什么是 Monorepo？**

```
[多仓库（Multi-repo）vs 单仓库（Monorepo）]

Multi-repo（传统方式）:
repo-ui-library/
repo-web-app/       ← 依赖 ui-library
repo-mobile-app/    ← 依赖 ui-library
repo-docs/

问题: 修改ui-library，需要发布npm包，其他repo再更新依赖，流程繁琐

Monorepo（现代方式）:
packages/
├── ui-library/     ← 共享组件库
├── web-app/        ← 依赖 ui-library（直接引用源码）
├── mobile-app/     ← 依赖 ui-library
└── docs/
package.json        ← 根包管理

优点: 代码共享简单，原子性提交，统一工具链
工具: pnpm workspace / Nx / Turborepo
```

### 6.5 设计模式在前端的应用

```javascript
// === 观察者模式（事件总线）===
class EventBus {
  constructor() {
    this.listeners = new Map();
  }
  
  on(event, callback) {
    if (!this.listeners.has(event)) {
      this.listeners.set(event, []);
    }
    this.listeners.get(event).push(callback);
  }
  
  emit(event, data) {
    this.listeners.get(event)?.forEach(cb => cb(data));
  }
  
  off(event, callback) {
    const cbs = this.listeners.get(event);
    if (cbs) {
      this.listeners.set(event, cbs.filter(cb => cb !== callback));
    }
  }
}

// === 工厂模式（创建不同类型的对象）===
class ButtonFactory {
  static create(type, props) {
    const ButtonMap = {
      primary: PrimaryButton,
      danger: DangerButton,
      ghost: GhostButton
    };
    const Button = ButtonMap[type] || DefaultButton;
    return new Button(props);
  }
}

// === 策略模式（表单验证）===
const validators = {
  required: (value) => value !== '' || '此项必填',
  email: (value) => /^[^\s@]+@[^\s@]+\.[^\s@]+$/.test(value) || '邮箱格式不正确',
  minLength: (min) => (value) => value.length >= min || `最少${min}个字符`,
  maxLength: (max) => (value) => value.length <= max || `最多${max}个字符`,
};

function validate(value, rules) {
  for (const rule of rules) {
    const result = rule(value);
    if (result !== true) return result;  // 返回错误信息
  }
  return true;
}

// 使用
validate(email, [validators.required, validators.email]);

// === 单例模式（全局唯一实例）===
class GlobalStore {
  static instance = null;
  
  static getInstance() {
    if (!GlobalStore.instance) {
      GlobalStore.instance = new GlobalStore();
    }
    return GlobalStore.instance;
  }
  
  private constructor() {
    this.data = {};
  }
}
```
